# 项目简报：基于 ZhihuRec 的搜索-推荐闭环系统

## 0. 核心目标

这个项目**不只是**一门课的作业，也不是一个玩具级的推荐系统 Demo。

真正的目标是打造**一个核心项目，产出四份成果**：

1. 一份**可视化数据分析报告**
2. 一份**HCI 报告**
3. 一个看起来像真实搜索/推荐系统的**代码项目**
4. 一个由以上三者沉淀出的、可写进简历的**端到端项目故事**

对于 `V1`，首要目标应明确为：

> 一个面向简历过筛与面试讲述的工程项目

课程交付仍然重要，但应服务于这条主线，而不是压过工程项目本身。

这个项目应该帮助你：

- 通过这学期的两门课程
- 产出一个足够扎实、适合面试讲述的搜索/推荐系统项目
- 支撑用一条简历 bullet 或一个面试 hook 讲清完整系统链路与 trade-off
- 支撑搜索、广告、推荐、排序、召回、平台基础设施等方向的实习 / 校招面试

---

## 1. 项目定位

### 主轴

这个项目的主要定位应该是：

> 一个真实感较强的内容社区搜索 + 推荐闭环系统

在 `V1` 阶段，这个定位应进一步收敛为：

> 一个以工程闭环和面试可讲述性为优先目标的搜索 / 推荐项目

### 学术 / 讲故事切入点

这个项目的故事切入点是：

> 当用户从推荐流切换到搜索时，往往意味着当前推荐尚未完全捕捉用户此刻的真实意图。
> 因此，搜索应该被视为一种高意图信号，并反哺推荐系统。

需要明确区分两层含义：

- 这条叙事是项目在面试中的**故事 hook / 谈资**
- 它不是在实现层面取代推荐主链路成为新的工程主体

更直白地说：

> `A` 是工程实现主线，`B` 是面试表达主 hook

也就是：

- 工程上，项目仍以推荐系统主链路为主体
- 讲述上，最有记忆点的部分是“搜索如何作为高意图信号反馈到后续推荐”

这个切入点主要用于：

- HCI 叙事框架
- 数据分析叙事框架
- 面试中的项目讲述

### 工程优先级

这个项目在工程上应优先被构造成：

- 一个真实的推荐流水线
- 然后是搜索对推荐的反馈
- 最后是事件级的闭环

这可以理解为：

- 推荐主链路决定项目是否站得住
- 搜索反馈机制决定项目是否足够有记忆点、足够好讲

这里的“真实”指的是：

- 有离线数据处理
- 有在线服务接口
- 有数据库持久化
- 有事件写回
- 有一个足够调试和演示用的极简前端

### V1 后端服务形态

`V1` 的后端应采用：

> 逻辑单体 + 模块分层

也就是说：

- 整体上仍是一个后端服务
- 但代码结构上应清晰划分 `recommendation`、`search`、`profile`、`event` 等模块
- 目标是既保持第一版落地简单，又保留足够强的工程叙事

在这个边界下，`V1` **不要**一开始引入：

- 消息队列
- 服务注册 / 服务发现
- 微服务拆分
- 以削峰填谷为目标的复杂异步架构

原因很明确：

- 第一版重点不是吞吐治理和分布式复杂性
- 第一版重点是把搜索、推荐、画像、事件回写做成一个可运行、可解释的闭环
- 逻辑单体已经足够支撑当前阶段的工程实践和面试讲述

优先级顺序：

1. **推荐主流程**
2. **搜索反馈进入推荐**
3. **用户反馈闭环**

## 2. 业务场景

**不要**再沿用之前那个多语言词汇平台作为项目外壳。

应改用下面这个场景：

> 一个类似知乎的内容社区。用户浏览推荐流，当推荐不足以满足需求时会发起搜索；而搜索行为又会进一步反馈到之后的推荐中。

这个项目可以借鉴**黑马点评**的后端工程思路，但**不能**继续停留在本地生活 / 店铺点评项目上。

### 可借鉴黑马点评的部分

可以借鉴：

- 用户系统
- 内容 / Feed 服务风格
- Redis / MySQL 的工程习惯
- 交互日志的记录思路
- 后端骨架 / 模块化风格

**不要**把重点放在：

- 优惠券秒杀
- 地理位置附近搜索
- 本地商家订单
- 本地生活业务逻辑

这个项目是**内容分发系统**，不是本地商业系统。

---

## 3. 数据集策略

### 主数据集

使用 **ZhihuRec** 作为唯一的主数据集基础。

ZhihuRec 会被用于三个目的：

1. **研究层**

   - 可视化数据分析
   - HCI 行为分析
   - 推荐转搜索现象分析

2. **建模层**

   - 构建用户兴趣信号
   - 构建 query-topic 映射
   - 构建内容 / 回答表示
   - 支持召回 / 排序逻辑

3. **演示数据基础**
   - 为 Demo 重建一个规模较小但足够真实的内容世界
   - **不要**在前端直接暴露完整原始数据集
   - 而是抽样 / 重建一个更像产品环境的小型内容世界

### V1 数据规模边界

`V1` 应明确采用：

> 小而精的数据规模策略

也就是说：

- 在线演示内容世界不追求大规模
- 优先保证内容实体、topic 关系、召回链路、画像更新和闭环逻辑都足够清楚
- 优先保证数据可调试、可查看、可解释

这个项目在第一版中不需要通过“很大的数据量”来证明价值。
第一版真正要证明的是：

- 你能把数据处理、召回、排序、事件回写、画像更新串成完整系统
- 你能解释系统为什么这样设计
- 你能把 trade-off 讲清楚

因此，数据规模应服务于工程闭环，而不是反过来让规模绑架实现复杂度。

### 重要原则

ZhihuRec **不只是**一个前端内容来源。

它应该扮演：

- 真实行为模式的来源
- 内容实体及其关系的来源
- 真实日志与 Topic 结构的来源

## 4. 核心产品结构

### 推荐主实体

主要的推荐实体应该是：

> **Answer**

### 辅助实体

- **Question**：回答的父级上下文
- **Topic**：连接 query、内容与用户兴趣的中间语义层
- **Author**：内容生产者
- **Query**：用户主动搜索时表达的意图
- **User**：拥有画像与行为历史的内容消费方

### 为什么以 Answer 作为主要推荐实体

原因是：

- 更接近用户真实消费对象
- 能支持更细粒度的推荐与反馈建模
- 更适合 Feed 卡片式展示
- 更容易调试和解释

### 产品展示策略

`V1` 前端在展示形态上应采用：

> 控制台式调试前端

其核心区域仍应以单列的 Answer 卡片流为主，但是否单页不是关键，关键是最利于后端调试与链路观察。

**不要**在第一版就把产品界面做得太精致。
前端首先应被视作：

- 调试控制台
- 推荐可观测性面板
- 闭环可视化工具

---

## 5. 前端理念

### 重要原则

这是一个由单个开发者完成的练习项目。
前端**不需要**为了真实用户而做得非常精致。

对 `V1` 来说，前端应明确采用：

> 能简则简

它的职责不是承担产品体验，而是最低成本地支撑后端开发、调试、演示和面试讲述。

更明确地说：

- 前端的首要作用是服务后端开发
- 前端不是为了真实用户易用性而设计
- 只要能把接口、召回结果、画像变化和事件闭环展示清楚，就已经达标

前端应该优先满足：

- 最低开发成本
- 最容易调试
- 最清晰地展示推荐决策
- 足够支撑面试时把系统链路讲完整
- 足够方便后端开发时直接观察系统行为

### 因此

应尽可能在前端直接暴露有用的调试信息。

每张 Answer 卡片可以展示：

- 问题标题
- 回答摘要
- 作者
- Topics
- 召回来源
- 召回分数
- 最终分数
- 是否触发 fallback
- 被选中的原因
- 匹配到的最近 query
- 用户兴趣匹配情况

在内容展示层面，`V1` 的 Answer 卡片应明确收敛为：

> 标题 + 摘要 + author + topics

也就是说：

- 不把第一版卡片继续扩展成更重的详情型组件
- 先保证内容信息足够支撑调试、解释和演示
- 详情展开能力不是 `V1` 的前提

在这些字段中，`V1` 的推荐 Feed 调试信息应至少强制展示：

- 召回来源
- 基础召回分
- `topic` 匹配分
- `query recall boost`
- 最终分数
- `fallback` 标记
- 被选中原因

其他字段可以作为补充展示，但这组字段应视为 `V1` 的必显集合。

其中，`被选中原因` 在 `V1` 中应明确采用：

> 规则模板生成

也就是说：

- 不做复杂自然语言生成
- 也不只是机械展示命中标签
- 应根据最高贡献项或主要贡献组合，生成一句简洁、可解释的原因说明

例如可以是：

- 因 `dual_tower` 召回分较高而进入前列
- 因 `topic` 匹配与最近搜索 boost 被提升
- 因主召回不足而由 `hot_fallback` 补齐

### V1 页面形态

`V1` 应采用：

> 轻前端 + 适度模块化

也就是说：

- 不追求复杂前端架构
- 推荐 Feed、搜索输入、搜索结果、用户画像摘要、调试信息可以按最利于调试的方式组织
- 可以是单页，也可以是少量简单页面；是否单页不是硬约束
- 更重要的是把 feed、search、profile、debug panel 拆成清晰模块
- 页面布局服务于开发调试效率，而不是服务于用户操作路径优化

### 推荐的首页设计

一个偏调试风格的 Feed 页面，展示：

- 当前用户画像摘要
- 带调试字段的推荐卡片
- 搜索输入与结果区域
- 点击等最小必要交互
- 可选的详情查看入口

这个前端本质上是：

> 一个服务后端开发的推荐系统控制面板，而不是一个打磨精致、追求易用性的产品 UI

## 6. 用户画像设计

### 重要澄清

用户画像**不等于**用户 embedding 向量。

用户画像应理解为：

> 系统为一个用户维护的全部结构化信号

embedding 向量只是画像中的一种紧凑表示。

### V1 的用户画像应包含

- `topic_weights`
- `recent_clicked_answers`
- `recent_queries`

这个选择很重要，因为它支持：

- 可解释性
- 调试
- 面试中的项目讲述
- 后续搜索反馈整合

### 为什么采用这种画像结构

- `topic_weights` = 中期、可解释的兴趣状态
- `recent_clicked_answers` = 最近、具体的消费行为
- `recent_queries` = 最近的高意图但相对更弱的搜索意图信号

对于 v1 来说这已经够了。
**不要**把画像设计得过度复杂。

其中，`topic_weights` 在 `V1` 中应明确为：

- 只保留权重最高的 Top `10` topics
- 主要服务于兴趣解释、召回增强和前端调试展示
- 不在第一版中维护过大的 topic 权重集合

同时，`topic_weights` 的更新策略应明确采用：

> 加法累积 + 全局衰减

也就是说：

- 每次发生会影响画像的点击事件时，先对现有 `topic_weights` 做一次统一衰减
- 再根据本次事件命中的 topics 做加分累积
- 最后只保留 Top `10` topics

这样做的原因是：

- 保留兴趣随行为累积的直觉
- 避免很早的历史行为永久主导画像
- 让画像能随着最近行为逐步漂移
- 在第一版里仍然保持实现简单、可解释

对于 `recommendation_click`，`V1` 应进一步明确为：

> 直接依据被点击 answer 自带的 topics 更新 `topic_weights`

也就是说：

- 不再额外引入 question topics 补充逻辑
- 不做二次 topic 映射再更新
- 第一版优先保持更新路径短、清晰、可解释

对于 `search_result_click`，`V1` 应进一步明确为：

> 组合使用 `query -> topic` 命中的 topics 与被点击 answer 自带的 topics 更新 `topic_weights`

也就是说：

- 不是只看 query topics
- 也不是只看 answer topics
- 两者组合后的重合部分，应被视为更强确认信号

这里同样应保持一个明确边界：

> 只规定“重合部分加得更多”，不在 `V1` brief 中写死具体倍数或公式

也就是说：

- 交集 topics 的更新强于非交集 topics
- 但精确权重关系留给实现配置与测试调优
- 不在第一版文档中提前锁死 `2x`、`3x` 或更复杂规则

这样做的原因是：

- query 暴露了用户主动意图
- answer click 完成了内容层面的偏好确认
- 两者合起来更符合“搜索先表达意图、点击再确认意图”的闭环逻辑

其中，`recent_clicked_answers` 在 `V1` 中应明确为：

- 只保留最近 `10` 条点击记录
- 每条记录保留最小必要时间信息
- 主要服务于用户向量构造与近期行为解释

其中，`recent_queries` 在 `V1` 中应明确为：

- 只保留最近 `5` 条 query
- 主要服务于轻量意图记录与 topic-aware boost
- 不把第一版做成完整的搜索历史管理系统

### V1 画像存储形态

`V1` 的用户画像应采用：

> 一张用户画像表 + 偏结构化字段存储

也就是说：

- 不把画像拆成很多张明细表
- 优先使用一张 `user_profile` 或等价主表承载核心画像
- `topic_weights`、`recent_clicked_answers`、`recent_queries` 可以使用 JSON、文本或其他便于快速落地的结构化字段存储

这样做的原因是：

- 第一版实现最简单
- 最容易同步直写更新
- 最容易调试和直接查看画像内容
- 最符合当前“闭环优先、复杂度靠后”的项目边界

后续如果画像规模、更新频率或查询模式变复杂，再考虑拆分成更细的画像表或明细表。

## 7. 冷启动 / 默认画像

由于系统从 ZhihuRec 起步、且没有真实在线用户，v1 应使用：

> **默认冷启动画像**

这比把它叫做“平均用户画像”更合适。

### 目的

它应当充当：

- 全局先验
- 新用户的起点
- 当用户特定信号稀疏时的一层轻量基础

### v1 冷启动策略

使用：

- 全局 topic 分布
- 基于 topic 的热门内容

之后可以再和用户特定信号进行混合。

`V1` 的默认画像与个性化画像应明确采用：

> 线性混合

也就是说：

- 在用户行为很少时，默认冷启动画像占更高权重
- 随着用户点击、搜索结果点击等有效行为增多，个性化画像权重逐步上升
- 默认画像权重逐步下降，而不是在某一个时刻被硬切掉

这样做的原因是：

- 比“有了一个行为就完全切到个性化”更稳
- 更符合冷启动到个性化过渡的直觉
- 更容易在面试中解释默认先验如何逐步退场

用于驱动这条线性混合的，不应只是单一事件数，而应是：

> 综合行为分数 `behavior_score`

在 `V1` 中，`behavior_score` 只由三类核心事件组成：

- `search_query`
- `recommendation_click`
- `search_result_click`

并且三者权重应满足：

- `search_query` = 低权重信号
- `recommendation_click` = 中权重信号
- `search_result_click` = 高权重信号

也就是说：

- `search_query` 可以直接进入冷启动混合权重的计算
- 但它只作为较弱的意图信号参与
- 真正更快推动系统从默认画像转向个性化画像的，仍然是点击类行为，尤其是 `search_result_click`

对于 `behavior_score -> alpha` 这一步，`V1` 应进一步明确为：

> 只保留相对趋势，不在 brief 中写死公式

也就是说：

- 只需要明确“行为越多、行为越强，个性化权重越高”
- 不在 `V1` brief 中提前锁死精确函数形式
- 具体计算方式在实现中应保留为可调参数或可替换逻辑

同时需要明确一个很重要的工程与面试原则：

> 参数不是拍脑袋定的，而是通过测试和调优逐步得到的

这意味着：

- `behavior_score` 的各事件权重应支持实验和调参
- 默认画像与个性化画像的混合参数应支持测试验证
- 最终保留下来的参数，应能在面试中解释为“基于测试结果和工程观察收敛得到”，而不是随意指定

`V1` 在参数调优验证上应优先采用：

> 离线回放测试

也就是说：

- 选取固定的单用户行为序列进行回放
- 观察不同参数下画像、召回结果和排序结果如何变化
- 用这套回放结果支撑参数选择与工程调优

`V1` 的离线回放测试场景，至少应覆盖：

- 推荐点击场景
- 搜索但不点击场景
- 搜索并点击结果场景

这样设计的原因是：

- 能同时覆盖中等信号、弱信号和强信号
- 能验证为什么 `search_query` 不能直接强更新画像
- 能验证为什么 `search_result_click` 应高于 `recommendation_click`

这样做的原因是：

- 最符合当前单用户闭环项目的形态
- 最容易验证“行为变化是否真的驱动了推荐变化”
- 最容易在面试中说明参数设计依据来自测试而不是直觉

这类参数调优在 `V1` 中的核心优化目标，应明确为：

> 响应性与稳定性的平衡

也就是说：

- 一次高意图行为之后，下一轮推荐应明显朝新意图方向变化
- 但系统又不能因为一次行为就发生过度漂移

对于“用户是否更满意”这件事，`V1` 应保持严谨表达：

> 不直接宣称已经在线证明用户满意度提升，而是通过满意度代理指标证明推荐结果更贴近用户当前意图

更适合 `V1` 的量化代理指标可以包括：

- `Intent Responsiveness@K`
  含义：高意图行为发生后，下一轮 Top-K 推荐中，与目标 topic 对齐的内容占比是否明显上升
- `Search Carryover Gain@K`
  含义：`search_result_click` 之后，下一轮推荐是否更快出现与该搜索 topic 相关的内容
- `Overreaction Penalty`
  含义：一次事件后，推荐列表是否发生过度漂移；如果变化过猛，则该项变差
- `Top-K Stability`
  含义：在没有新的强信号时，相邻两轮推荐结果应保持一定稳定性，而不是无意义抖动

如果需要更工程化地补充，还可以记录：

- `Fallback Exposure Rate`
  含义：推荐结果里 fallback 内容占比是否过高；这个指标更偏系统健康度，而不是直接满意度

在这些指标中，`V1` 的指标口径应进一步明确为：

> 主指标采用“响应性 + 稳定性”的组合视角

也就是说：

- 不把单一指标当成唯一目标
- 参数选择的主判断标准，是系统是否在“足够响应新意图”和“不过度漂移”之间取得平衡

同时还应明确一个关键次指标：

> `Search Carryover Gain@K` 是服务项目故事线的重点指标

原因是：

- 它最直接对应“搜索反哺推荐”这条核心 hook
- 它最容易向面试官解释：搜索点击之后，后续推荐到底有没有真正承接搜索意图
- 它应作为主指标之外、最重要的故事型辅助指标存在

因此，面试中的表述最好是：

> 我们通过离线回放测试对参数做 sweep，主看响应性与稳定性的平衡，同时重点观察 `Search Carryover Gain@K`，用它来支撑“搜索意图是否真正传导到了后续推荐”这条核心故事，而不是拍脑袋选参数。

这条规则的目的，是让系统既能感知“用户已经开始主动表达意图”，又不会因为一条 query 就过快抛弃默认先验。

重要的是：
不要过度设计冷启动。
它只需要回答两个问题：

1. 当用户没有个人历史时，系统展示什么
2. 系统如何从默认画像逐步过渡到个性化画像

### V1 用户范围边界

`V1` 应明确采用：

> 单用户闭环演示

也就是说：

- 系统围绕一个默认用户或演示用户跑完整闭环
- 不做完整登录系统
- 不做多用户切换和多用户对比展示

这样做的原因是：

- 最容易把“这次行为如何改变下一轮推荐”讲清楚
- 最容易调试画像更新、事件回写和结果变化
- 最符合当前“第一版尽量简单，但闭环必须成立”的目标

如果后续需要增强演示性，再考虑加入“重置画像”或少量预设用户，但这不是 `V1` 的前提。

## 8. 推荐架构（v1）

### 推荐亮点

这个核心项目的亮点应该是：

> 具备双塔召回 + ANN + fallback 的推荐流水线

这里的“双塔召回”在 `V1` 中应被理解为：

> 重点保证“离线向量产出 + 在线 ANN 检索 + 工程可落地”，而不是把训练本身做得很重

### 双塔实现边界

`V1` 对双塔的真实性要求应落在：

- 有真实的用户 / 内容向量产出流程
- 有真实的向量索引构建
- 有在线 ANN 查询或等价的近邻召回服务
- 有可观测的召回结果、分数和来源

但同时要明确：

- 训练过程不是 `V1` 的重点
- 不要求把双塔训练本身做成一个很重的建模项目
- 必要时甚至可以不做完整训练，而是直接使用离线生成的表示或轻量方案支撑工程链路

换句话说：

> `V1` 要证明的是“你把向量化召回系统真正落地了”，而不是“你把模型训练做得多复杂”

### V1 向量来源策略

`V1` 在向量来源上应明确采用：

> 内容向量预计算，用户向量规则聚合

也就是说：

- `Answer` 向量离线生成
- `User` 向量在线由 `recent_clicked_answers` 和 `topic_weights` 聚合得到
- `recent_queries` 不直接进入用户向量，而是在召回或排序阶段作为轻量 boost 使用

### V1 Answer 向量组成

`V1` 的 `Answer` 向量应由以下字段共同生成：

> `question_title + answer_summary + topics`

这样做的原因是：

- 比只用标题或摘要更完整
- 比直接引入全文更轻、更稳
- topic 信息能帮助向量表示和后续解释保持一致

这意味着第一版**不需要**：

- 引入完整 answer 全文做主向量
- 维护多套长短文本向量
- 为内容向量设计过重的字段拼接策略

这一策略的好处是：

- 最符合当前项目“训练不重、工程链路要真”的边界
- 能把向量召回、用户画像、搜索反馈三者自然串起来
- 面试里容易讲清系统链路和 trade-off

### V1 用户向量构造

`V1` 的用户向量应进一步明确为：

> `recent_clicked_answers` 向量加权平均 + `topic_weights` 原型向量

这意味着：

- 用户最近点击过的 answer 向量构成主要的短期兴趣表示
- `topic_weights` 提供中期、可解释的兴趣先验
- 两者组合形成在线用户表示，用于 ANN 召回
- 来自 `search_result_click` 的 answer 也应进入 `recent_clicked_answers`，并以更高权重参与这一步聚合

这里的“更高权重”同样应理解为：

- 相对顺序先固定
- 精确倍数后置到实现配置

两者的相对关系应明确为：

> 点击向量为主，topic 原型为辅

也就是说：

- `recent_clicked_answers` 是用户向量的主体
- `topic_weights` 负责提供稳定但较轻的兴趣修正
- 用户最近行为应比静态或中期兴趣先验更能主导当前召回结果

同时，`recent_clicked_answers` 的聚合应明确采用：

> 时间衰减

也就是说：

- 越新的点击，对用户向量影响越大
- 越旧的点击，权重应逐步降低
- `recent_clicked_answers` 中的每条记录都应保留时间信息，以支持在线聚合时的衰减计算
- `V1` 中只使用最近 `10` 条点击记录参与这一步聚合

这里要继续保持一个明确边界：

- `recent_queries` 不直接进入用户向量
- 搜索 query 的作用仍主要体现在 topic-aware boost，以及搜索点击后的强权重画像更新上

### 面试表述边界

对于这套实现，最稳妥的表达不是：

> 完整训练了一个双塔模型

而更准确的表达应是：

> 实现了 `dual-tower-style vector recall`，即向量化双侧召回链路：离线生成内容向量，在线聚合用户表示，并通过 ANN 做候选召回

这样表述的好处是：

- 不会夸大训练部分
- 但能准确体现工程实现深度
- 与当前 `V1` 的真实实现边界一致

### v1 的召回桶

一开始只使用两个召回桶：

1. `dual_tower`
2. `hot_fallback`

原因：

- 最小化但足够真实的闭环
- 容易解释
- 容易调试
- 能覆盖冷启动和失败场景

在候选规模上，`V1` 应进一步明确为：

> `dual_tower + ANN` 先召回 Top `20` 候选，再排序到 Top `10`

这样做的原因是：

- 足够体现“召回”和“排序”是两层不同作用
- 第一版实现仍然足够轻
- 面试里也容易讲清候选集是如何被压缩的

其中，`hot_fallback` 在 `V1` 中应明确为：

> 全局热门 answer

也就是说：

- 不再细分成 topic 内热门或多级 fallback
- 第一版默认使用全局热门内容作为兜底来源
- 这样最简单，也最利于解释系统在主召回不足时如何补齐结果

这个“全局热门”应进一步理解为：

> 基于 ZhihuRec 的全局交互热度，在某个固定时刻截取的一份快照

也就是说：

- 目的是模拟某个时刻的真实线上热门状态
- 不在第一版里继续纠结更复杂的热门定义
- 重点是让 fallback 足够真实，能支撑闭环，而不是把热门逻辑本身做成研究点

`V1` 的 fallback 触发条件也应明确为：

> 只要主召回数量不足 Top `10`，就用 `hot_fallback` 补齐

也就是说：

- 不额外引入分数阈值判断
- 不要求主召回完全为空才触发
- 第一版只按数量是否补满来决定是否兜底

### v1 不要一开始加太多召回桶

**不要**从下面这些开始：

- 单独的 topic-based bucket
- 单独的 search-based bucket
- 单独的 follow-based bucket
- 多阶段复杂 rerank 树

这些都可以后面再做。

### 为什么

第一目标是：

- 先让推荐主循环真实跑起来
- 先让 fallback 可见
- 先让系统行为可解释
- 先把向量召回的工程链路跑通，再考虑训练复杂度

## 9. 搜索反馈设计

### 主切入点

搜索应被视为：

> 用户具有更主动、更具体需求的信号，而这种需求可能尚未被推荐流充分满足

这一点在项目里的定位应明确为：

- 它是最核心的故事点和面试谈资
- 但它建立在推荐主链路已经成立的前提上
- 它是推荐系统的增强闭环，不是替代推荐系统本身

### 重要细节

**不要**过度宣称“搜索一定意味着不满意”。
更稳妥的解释是：

> 搜索是一种更高意图的信号，它经常反映尚未被满足、或尚未被系统捕捉到的需求

### V1 搜索模块边界

`V1` 的搜索模块应定义为：

> 一个极简但真实的搜索链路

它不是一个独立打磨的“大而全搜索产品”，而是在尽量简单的前提下，同时满足三件事：

1. 搜索侧的**召回机制本身能够成为亮点**
2. 能完整支撑“搜索反哺推荐”的故事
3. 能真实改动用户画像

因此，`V1` 搜索至少要具备：

- 一个真实可用的搜索输入入口
- 一个最小可用的搜索结果列表
- 一条可解释的搜索召回链路
- 搜索点击后的事件回写
- 搜索行为对后续推荐的实际影响

对于“输入 query”这件事，`V1` 应进一步明确为：

> 不在输入 UI 上过度投入，优先提供方便后端模拟真实输入的开发方式

也就是说：

- 不追求产品化的搜索输入体验
- 前端输入框只需要满足最小可用
- 更重要的是提供方便后端调试和回放的脚本化输入方式

`V1` 在这部分应优先提供：

> 命令行脚本，既支持单个 query 调试，也支持 query 序列回放

这样做的原因是：

- 同一套入口可以同时服务后端调试与离线回放
- 比单纯依赖前端手工输入更高效
- 更适合单用户闭环项目的工程验证方式

其中，`query` 序列回放的输入格式应优先采用：

> JSON 序列

并且应支持携带：

- 时间信息
- 动作类型

这样做的原因是：

- 后续可以自然扩展到 `search_query`、`search_result_click`、`recommendation_click` 等不同事件
- 比纯文本一行一个 query 更适合回放真实行为序列
- 更方便后续测试脚本、参数调优和事件重放复用同一套输入结构

`V1` 在动作粒度上，应至少支持：

- `search_query`
- `search_result_click`
- `recommendation_click`

这样正好与当前 `V1` 已经定义好的三类核心事件保持一致。

这类方式可以包括：

- 开发脚本触发搜索请求
- 固定 query 序列回放
- 便于模拟真实输入流的调试入口

核心目标不是把“输入框”做精致，而是让你能低成本模拟真实 query 输入并观察系统反应

### V1 搜索结果来源策略

`V1` 的搜索结果来源应采用：

> `query -> topic -> answer` 为主，`hot_fallback` 只用于补齐不足结果

也就是说：

- 搜索主结果应来自 query-topic-answer 这条相关性链路
- 热门内容不是搜索主排序逻辑，只在结果不足时作为兜底补齐
- 默认不采用“similarity 分数 + 热门分数直接统一加权混排”作为 `V1` 主策略

更具体地说，`V1` 更适合：

- 先按相关性链路召回并排序搜索候选
- 如果相关候选不足，再从 `hot_fallback` 补充剩余结果位
- 在结果和调试信息中明确标识每条内容的来源

这样设计的原因是：

- 更符合搜索场景下“相关性优先”的直觉
- 更容易保证“搜索反哺推荐”的故事不被热门逻辑冲淡
- 更容易调试和解释为什么某条结果会出现
- 更适合在面试中讲成清晰的 bucket + fallback 机制

如果后面确实需要在少量结果位中引入热门信号，也应优先采用：

- 分桶后补齐
- 或者对热门结果施加强 penalty / 强配额限制

而不是在第一版里直接把相关性和热门做成等价的混合加权。

这里的“可解释搜索召回链路”应尽量体现为：

> query -> topic -> answer

也就是说，搜索的亮点不在于复杂排序，而在于：

- query 如何被映射到 topic
- topic 如何召回到 answer
- 这些信号如何再反馈回推荐系统

为了保持 `V1` 简洁，搜索模块**不要**一开始加入：

- 复杂筛选器
- 多阶段搜索排序
- Query Rewrite / 扩写纠错等复杂查询理解
- 历史搜索、联想词、个性化搜索页等产品化细节

### 信号强弱层级

将用户信号分层看待：

#### V1 最小事件集中的弱信号

- 原始搜索 query

#### V1 最小事件集中的中等信号

- 推荐点击

#### V1 最小事件集中的强信号

- 搜索结果点击

如果从完整系统角度看，未来仍可以继续保留下面这类更强信号作为扩展：

- like
- favorite
- 对相似内容的重复点击

对 `V1` 来说，这里的关键不是把所有事件都做全，而是把三类核心信号的相对强弱讲清楚：

- `search_query` = 弱信号
- `recommendation_click` = 中等信号
- `search_result_click` = 强信号

这样定义的原因是：

- `search_query` 只表达意图，不代表偏好确认
- `recommendation_click` 代表用户对被动曝光内容产生兴趣，但意图强度仍弱于主动搜索后的点击
- `search_result_click` 发生在用户主动表达 query 之后，更适合作为强确认信号
- 因此，`search_result_click` 在画像更新和用户表示构造中的权重都应高于 `recommendation_click`

这里的要求首先是：

> 保证相对强弱顺序，而不是在 `V1` brief 中写死具体倍数

也就是说：

- 只需要明确 `search_result_click_weight > recommendation_click_weight`
- 具体数值在实现中应作为可配置参数保留
- `V1` 文档不预先锁死成 `2x` 或 `3x`

### v1 的搜索更新规则

使用下面这条规则：

- 当用户输入搜索 query 时：

  - 写入 `recent_queries`
  - **不要**直接更新长期 `topic_weights`

- 当用户点击某个搜索结果时：
  - 再更新 `topic_weights`

在 `V1` 中，更具体的更新顺序应是：

1. 先对现有 `topic_weights` 做全局衰减
2. 再按本次事件对应的 topics 做加法累积
3. 最后截断到 Top `10`

这是一个有意做出的权衡：

- 原始 query = 更轻、更噪的意图信号
- 推荐点击 = 中等强度的兴趣确认
- 搜索点击 = 更强的偏好确认

在 `V1` 中，这里的“更强”应首先体现为：

- `search_result_click` 的更新权重高于 `recommendation_click`
- 具体倍数不在 brief 中写死，而留给实现阶段配置

这点应在项目说明里明确写清楚。

## 10. Query 到 Topic 的映射

### query-topic mapping 在做什么

它做的事情是把：

> 用户临时性的表述方式 -> 系统内部稳定的 topic 空间

换句话说：

- 用户通过 query 文本表达意图
- 系统需要一个稳定的内部语义层
- topic 充当搜索信号和推荐内容之间的桥梁

目标链路是：

> query -> topic -> answer

### v1 的要求

这种映射必须满足：

- 足够真实
- 低延迟
- 低成本
- 易于调试
- 易于在面试中解释

### V1 Topic 空间边界

`V1` 的 Topic 空间应采用：

> 保留 ZhihuRec 原生 topic 体系，但只做轻度清洗 / 合并，并只取一个可控子集

也就是说：

- 不在第一版重新设计一整套新的 topic taxonomy
- 先检查原生 topic 的数量、分布和可用性
- 再选择一个足够支撑闭环和面试讲述的 topic 子集
- 只做必要的轻清洗、轻合并、去噪和去异常 topic 处理

这个边界的核心目的是：

- 保留数据真实性
- 避免 topic 体系治理本身变成一个大工程
- 让 `query -> topic -> answer` 这条链路足够稳定、足够好讲
- 让第一版把重点放在闭环跑通，而不是放在 topic 设计本身

因此，topic 选择原则应优先考虑：

- 高频且语义相对稳定的 topic
- 能覆盖主要内容与 query 场景的 topic
- 数量可控、便于调试和观察的 topic

### 最终决策

使用：

> **基于 ZhihuRec 离线统计得到的 query-topic 共现关系**
>
> - 在线阶段通过倒排映射做轻量级 keyword / token 查找

v1 **不要**使用在线 embedding 相似度。

这里隐含的前提是：

- query-topic 映射只在选定的 topic 子集上构建
- topic 子集应尽量稳定、干净、可解释
- 如果原生 topic 过多或过散，先裁子集，再做映射统计

### 为什么不先上 embedding

因为 v1 的优先级是：

- 延迟
- 低工程成本
- 可解释性
- 可调试性
- 简历 / 面试表达清晰

基于 embedding 的语义匹配可以明确写成：

> 面向更强语义泛化能力的后续扩展

### 在线策略

运行时：

- 对 query 做归一化 / 分词
- 查 query-topic 映射
- 按配置文件中的动态阈值选取相关 topics，而不是写死固定 Top N
- 基于这些 topics 找到相关 answer 候选
- 优先返回相关性链路得到的搜索结果
- 如果相关候选不足，再用 `hot_fallback` 补齐
- 返回固定 Top `10` 的最小可用且可解释的搜索结果列表
- 将这些 topics 作为轻量级召回增强信号

如果前端暴露调试信息，搜索结果最好也能展示：

- query 命中的 topic
- 结果命中的 topic / answer
- 搜索结果来源或命中原因
- 每条结果的匹配分组成

`V1` 在搜索结果调试展示上，应至少满足：

> topics + 来源 + 匹配分组成

这样做的原因是：

- 能直接解释“为什么是这条结果被召回”
- 能把 query-topic-answer 这条链路讲清楚
- 能服务后端调试和面试演示

### V1 query-topic 选取边界

`V1` 中，`query -> topic` 的在线选取应采用：

> 动态阈值 + 配置文件驱动

也就是说：

- 不在 brief 中写死一次必须取 Top `1` / `3` / `5`
- 配置形式优先采用“分数阈值 + 最大 topic 数”
- 具体阈值规则放在配置文件中
- 实现时应支持根据 topic 分数分布动态决定保留多少个 topic

这样做的原因是：

- 比固定 Top N 更灵活
- 更符合后续参数调优与回放测试的需要
- 面试里可以说明“topic 选取规则是可配置、可验证的”，而不是硬编码常数

在当前 `V1` 边界下，这里的 `score` 也应优先理解为：

> 基于离线 `query-topic` 统计关系的匹配分数，而不是在线 embedding 余弦相似度

也就是说：

- 在线阶段优先走更快的倒排查找 / 共现匹配分数
- 不把 query-topic 映射做成在线 embedding 检索问题
- embedding 更适合保留在 answer 向量召回层，而不是优先放到 query-topic 这一层

更具体地说，`V1` 的 `query -> topic score` 应采用：

> 离线共现分数聚合 + 少量规则加成

也就是说：

- 每个 query token 或短语先从离线映射表中取出对应的 topic 候选与基础分数
- 在线阶段对这些基础分数做聚合
- 在此基础上允许加入少量规则加成

这些规则加成应保持克制，优先考虑：

- 完整短语命中高于单 token 命中
- 多 token 一致指向同一 topic 时可获得额外加成

`V1` 在这部分应明确收敛为：

> 只保留两条规则加成

也就是说：

- 保留“完整短语命中高于单 token 命中”
- 保留“多 token 一致指向同一 topic 时加成”
- 暂不继续扩展更多规则系统

这一层的设计目标是：

- 保留统计映射的主线
- 让 query-topic 匹配更像真实搜索工程
- 但仍然保持第一版的实现简单、可配置、可调试

这样做更符合当前项目的要求：

- 更快
- 更便宜
- 更容易调试
- 更容易解释
- 不违背“`V1` 不做在线 embedding query-topic 映射”的既有边界

### 存储策略

最终目标架构：

- MySQL 作为 source of truth
- Redis 作为缓存 / 预热层

但实现顺序应该是：

1. `V1` 先只用 **MySQL** 跑通
2. Redis 明确放到第一版跑通之后再加
3. 如有需要，再把 Redis 作为缓存 / 预热层接入

这是有意为之。
系统应优先保证：

- 先有功能闭环
- 再做性能优化

更具体地说，`V1` 的边界是：

- 系统不能依赖 Redis 才能运行
- 不在第一版中处理缓存一致性、预热策略和失效策略
- Redis 只作为后续性能优化项保留

## 11. 推荐系统如何使用搜索信号

### 主流架构参考

更主流的工业推荐链路通常会区分：

- 召回：先决定哪些内容值得进入候选集
- 粗排 / 预排：快速压缩候选规模
- 精排：对更小候选集做更细打分

在这类架构里，`query` 这类高意图但弱确认信号，最自然的入口通常是：

> 先影响召回，再在系统更成熟时逐步进入粗排 / 精排特征

原因是：

- `query` 首先改变的是候选空间
- 它更适合表达“当前想看哪类内容”
- 它不应该在第一版里被过早做成复杂精排信号

### v1 设计

`recent_queries` 首先应该被用于：

> 通过 topic-aware 的过滤 / boosting 去影响召回

v1 **不要**把 recent queries 直接喂进双塔模型的用户塔。

更具体地说：

- `recent_queries` 不参与用户向量构造
- 它只通过 `query -> topic` 映射影响候选生成、召回增强和候选空间调整
- 真正进入用户向量主体的是 `recent_clicked_answers` 与 `topic_weights`

一开始也**不要**把搜索信号直接用于粗排 / 精排模型。

换句话说：

- `V1` 里，`search_query` 只影响召回层
- 不把 raw `search_query` 直接做成精排特征
- 也不在第一版里单独搭建更重的粗排 / 精排双模型体系

### v1 排序边界

`V1` 的排序层应明确采用：

> 规则型轻排序

也就是说：

- 不上单独的学习排序模型
- 不做两阶段复杂排序
- 排序逻辑应保持可解释、可调试、可展示

一个推荐的规则打分结构可以是：

- 双塔召回分数作为基础分
- `topic_weights` 匹配作为兴趣修正项
- `hot_fallback` 候选天然低于正常召回候选

这里要明确：

- `recent_queries` 对候选的影响发生在召回阶段
- 排序层可以展示 `query recall boost` 这类调试字段
- 但这不等于把 raw `search_query` 直接作为排序项使用

前端调试信息应尽量能展示每个候选的：

- 基础召回分
- topic 匹配分
- query recall boost
- fallback 标记
- 最终排序分

### 为什么

这样可以让第一版：

- 更简单
- 更容易验证
- 更容易调试
- 更容易解释
- 更容易在面试中讲清每个分数项的 trade-off

一个简单的打分直觉可以是：

- 双塔提供基础召回信号
- topic 匹配提供用户兴趣修正
- recent-query-to-topic 匹配提供轻量级召回增强，而不是直接精排打分

这已经足够构成一个有意义的第一版闭环。

## 12. 在线闭环（v1）

### V1 事件边界

`V1` 只持久化三类核心事件：

- `search_query`
- `recommendation_click`
- `search_result_click`

这样做的目的不是否认其他行为的重要性，而是为了在第一版中用最小复杂度支撑完整闭环：

- 有输入意图
- 有推荐消费反馈
- 有搜索后的强确认反馈

`impression`、`skip`、`like`、`favorite` 等行为可以保留为后续扩展，不作为 `V1` 的核心落盘前提。

### V1 画像更新策略

`V1` 的画像更新应明确采用：

> 同步直写画像 + 保留事件表

也就是说：

- 事件一发生，就直接更新数据库中的用户画像
- 同时保留原始事件表，作为可追溯记录和后续扩展基础

这样做的原因是：

- 第一版实现最简单
- 最容易把闭环讲清楚
- 最容易验证“这次行为如何影响下一次推荐”
- 仍然为后续异步化、重放、离线分析保留空间

因此，`V1` **不要**一开始把重点放在：

- 异步事件消费
- 复杂消息队列
- 画像重算流水线
- 近实时流式更新架构

### 推荐闭环

1. 用户打开 feed
2. 系统读取用户画像
3. 系统通过 `dual_tower` 召回答案候选
4. 如果候选不足 / 缺失，则使用 `hot_fallback`
5. 系统合并候选并执行规则型轻排序
6. 系统返回固定 Top `10` 的带调试字段答案卡片
7. 用户进行点击
8. `recommendation_click` 写入事件表，并同步更新 `recent_clicked_answers` 与 `topic_weights`
9. 下一轮推荐体现新的画像变化

### 搜索闭环

1. 用户输入 query
2. query 被写入 `recent_queries`
3. query 被映射到 topic
4. 搜索服务基于 topic 返回最小可用的 answer 结果列表
5. 后续推荐召回获得 topic-aware boost
6. 如果用户点击搜索结果
7. `search_result_click` 写入事件表，并基于 query 命中 topics 与 answer 自带 topics 的组合，以强于推荐点击的权重同步更新 `topic_weights`
8. 被点击的 answer 同时写入 `recent_clicked_answers`，并以更高权重参与用户向量聚合
9. 用户画像因 `recent_queries` 与搜索点击而发生变化
10. 后续推荐体现这一更强信号

这就是简历 / 面试中最核心的闭环故事。

## 13. 权衡原则

所有决策都应该围绕这个首要目标来评估：

> 让这个项目容易写进简历、容易在面试中解释，也容易围绕系统链路和工程权衡展开讨论

### 因此，优先选择：

- 简单但可解释的方案
- 真实但不过度堆砌的架构
- 可见的闭环，而不是复杂且隐藏起来的“高级感”
- 调试可见性，而不是打磨精致的 UI
- 能讲清完整链路与 trade-off，而不是堆积炫技点
- 工程落地优先于训练复杂度
- 同步直写的简单闭环优先于过早异步化
- 单用户可讲清的闭环优先于多用户产品化展示
- 逻辑单体的模块分层优先于过早微服务化
- 先把参数关系定清楚，再通过测试调优具体数值
- 分阶段演进，而不是一口气做成“大而全系统”

### 已经做出的示例权衡

- 以 Answer 为主要推荐实体，而不是更复杂的混合实体 Feed
- 使用轻前端 + 适度模块化的调试前端，而不是构建追求易用性的产品页面
- `V1` 只重建一个小而精的内容世界，而不追求大规模在线数据集
- 先做 `dual_tower + hot_fallback`，而不是很多召回桶
- 双塔优先保证“离线向量产出 + 真 ANN 检索”，而不是重训练流程
- `V1` 采用“内容向量预计算 + 用户向量规则聚合”，而不是完整重训练双塔
- 用户向量由 `recent_clicked_answers` 加权平均与 `topic_weights` 原型向量构成
- 排序层先采用规则型轻排序，而不是额外引入学习排序模型
- 搜索先做成“极简但真实的召回链路”，而不是完整搜索产品
- 搜索结果以相关性链路为主，热门只做补齐，而不是直接混合加权
- `V1` 只落盘 `search_query / recommendation_click / search_result_click` 三类核心事件
- `V1` 采用“同步更新画像 + 保留事件表”，而不是先做复杂异步事件架构
- `V1` 用户画像先采用单表 + 结构化字段存储，而不是拆很多张画像明细表
- `topic_weights` 采用“加法累积 + 全局衰减”，而不是只增不减或复杂在线学习
- 默认冷启动画像与个性化画像采用线性混合，而不是一刀切切换
- 冷启动混合权重由 `search_query / recommendation_click / search_result_click` 的综合行为分数驱动
- `behavior_score -> alpha` 只定义相对趋势，具体参数通过测试与工程调优确定
- `V1` 的参数调优验证优先采用离线回放测试，而不是只靠人工观察
- `V1` 保留原生 topic 体系，但只做轻清洗并选择一个可控子集
- `V1` 只围绕一个演示用户跑闭环，不做完整多用户系统
- `V1` 后端采用“逻辑单体 + 模块分层”，不引入消息队列和服务注册
- 使用 `topic_weights + recent clicks + recent queries`，而不是庞大的用户画像设计
- 先存原始 query，只有在搜索点击后才更新 topic
- 用共现统计 + 倒排查找做 query-topic 映射，而不是在线 embedding
- `V1` 先只用 MySQL，Redis 在第一版跑通后再加
- `V1` 只要求本地单机跑通，但必须提供一键初始化脚本
- 先跑通功能闭环，再做延迟优化

## 14. Codex 接下来应该帮助完成什么

Codex 现在应开始以结构化方式推进这个项目的落地。

### 眼下的任务

1. 定义准确的后端 schema
2. 定义最小化前端调试页面结构
3. 定义基于 ZhihuRec 的离线数据预处理流水线
4. 定义如何构建：
   - 默认冷启动画像
   - query-topic 映射表
   - 用户 topic 权重
   - answer 级推荐候选集
5. 定义 v1 的 API 契约
6. 定义最小化的推荐服务流水线
7. 定义闭环事件更新规则
8. 定义用于参数调优的最小测试 / 验证方案
   - 形式优先采用离线回放测试
   - 固定一组单用户行为序列
   - 至少覆盖推荐点击、搜索不点击、搜索点击三类场景
   - 对比不同参数下画像、召回、排序的变化
   - 定义响应性、稳定性和意图对齐的代理指标

### V1 运行与交付边界

`V1` 的运行方式应明确采用：

> 本地单机跑通

也就是说：

- 后端服务、本地数据库、前端页面在本机环境即可完成启动和演示
- 第一版不要求容器化交付
- 第一版不要求远端部署可访问

但同时必须满足：

> 要有一键初始化脚本

这个脚本至少应覆盖：

- 基础环境准备说明或自动化步骤
- 数据库初始化
- 必要的种子数据 / 演示数据准备
- 启动前所需的最小预处理步骤

此外，`V1` 还应提供：

> 开发脚本形式的一键重置能力

也就是说：

- 支持通过开发脚本重置单用户画像或演示状态
- 这项能力主要服务于调试、离线回放和面试演示
- 不需要在前端提供专门的“重置”入口

在搜索输入这件事上，也应采用同样思路：

- 优先提供开发脚本或调试脚本来模拟 query 输入
- 不要求依赖前端手工操作才能验证搜索链路
- 这类脚本应同时支持单 query 调试和 query 序列回放

这样做的原因是：

- 最符合当前“先跑通、先能演示”的目标
- 最能降低环境搭建摩擦
- 最方便后续自己反复重建环境和快速演示

### V1 API 边界

`V1` 的接口粒度应至少收敛为：

> `/feed` + `/search` + 拆开的事件接口

也就是说：

- `/feed` 负责返回推荐结果
- `/search` 一次请求内同时负责处理 query、记录 `search_query` / `recent_queries`，并返回搜索结果
- `search_query` 不再额外走统一事件上报，也不拆单独的前置记录接口，而是在 `/search` 中自然完成写入
- 点击类行为拆成更直白的接口，例如：
  - `/event/recommendation_click`
  - `/event/search_result_click`

这样做的原因是：

- 第一版调试更直接
- 接口语义更清楚
- 比统一 `event_type` 上报更方便演示和排查

第一版不继续拆更多面向产品化的细接口，先把最小闭环接口链路跑通。

在用户身份处理上，`V1` 应进一步明确为：

> 接口层仍保留 `user_id` 参数，但默认只围绕一个演示用户运行

也就是说：

- API 形态上允许传入 `user_id`
- 但第一版不做真正的多用户产品化使用
- 不做登录系统
- 不做 JWT 或完整鉴权链路

这样做的原因是：

- 保留一定工程合理性
- 但不把项目重心带到认证、会话和安全体系上
- 更符合“这些不是当前项目核心价值”的边界

### 给 Codex 的重要指令

不要过度设计系统。
除非某个额外的大组件能明显增强以下价值，否则不要加：

- 简历价值
- 闭环清晰度
- 面试可解释性

始终优先：

- 最小可运行的端到端流水线
- 明确的权衡
- 易于调试的实现
- 清晰的未来扩展点
- 先把工程链路跑通，再考虑是否补充训练复杂度
- 先用单服务把模块边界定义清楚，再决定未来是否需要拆分
- 先用 MySQL 跑通，再决定是否需要 Redis 优化
- 先提供本地一键初始化与启动能力，再考虑容器化和部署
- 关键参数先做成可调，再通过测试和工程观察收敛

## 16. 一句话内部总结

> 一个类似知乎的内容社区推荐系统，其中 Answer 是主要推荐实体，Topic 是连接 query 与推荐的桥梁，搜索作为高意图反馈信号，而第一版优先追求的是一个易调试、可解释的闭环，而不是功能面的大而全。

## 17. V1 边界总结

### 一句话版本

`V1` 的本质是：

> 一个面向简历过筛与面试讲述的、强工程闭环但极简前端的内容推荐系统；工程主线是推荐召回链路，故事 hook 是“搜索作为高意图信号反哺推荐”。

### 执行边界

- 项目目标：优先服务面试型工程项目，而不是研究型 Demo 或产品化 UI。
- 工程主线：`A` 是推荐主链路与向量召回工程落地，`B` 是“搜索反哺推荐”的面试故事。
- 推荐架构：`dual-tower-style vector recall + ANN + hot_fallback + 规则型轻排序`。
- 候选规模：`dual_tower + ANN` 先召回 Top `20`，再排序压到 Top `10`。
- fallback 来源：`hot_fallback` 明确采用全局热门 answer。
- 热门定义：基于 ZhihuRec 的全局交互热度，并截取一个固定时刻的快照来模拟线上状态。
- fallback 触发：只要主召回数量不足 Top `10`，就用 `hot_fallback` 补齐。
- 向量策略：内容向量离线生成，用户向量规则聚合；不做完整重训练双塔。
- Answer 向量：由 `question_title + answer_summary + topics` 生成。
- User 向量：`recent_clicked_answers` 加权平均为主，`topic_weights` 原型向量为辅；最近点击带时间衰减。
- 搜索定位：搜索是极简但真实的召回链路，核心是 `query -> topic -> answer`，不是完整搜索产品。
- 搜索输入：不在输入 UI 上过度投入，优先提供方便后端模拟真实 query 输入的脚本化方式。
- 搜索调试脚本：优先提供命令行脚本，既支持单 query 调试，也支持 query 序列回放。
- 回放输入格式：优先采用可携带时间和动作类型的 JSON 序列。
- 回放动作粒度：至少支持 `search_query / search_result_click / recommendation_click`。
- query-topic 选取：采用配置文件驱动的动态阈值，配置形式优先是“分数阈值 + 最大 topic 数”，而不是写死固定 Top N。
- query-topic score：优先使用离线统计得到的分数聚合，并允许少量规则加成，不走在线 embedding 余弦相似度。
- query-topic 规则：`V1` 只保留“完整短语优先”和“多 token 一致指向加成”两条规则。
- 搜索结果：相关性链路为主，热门只做补齐，不做等价混合加权。
- 搜索结果数：固定返回 Top `10`。
- 搜索调试展示：至少显示命中的 topics、结果来源和匹配分组成。
- 搜索信号用法：`search_query` 在 `V1` 只影响召回层，不直接进入粗排 / 精排。
- 推荐结果数：固定返回 Top `10`。
- 画像结构：单表 `user_profile`，包含 `topic_weights`、`recent_clicked_answers`、`recent_queries`。
- 画像规模：`topic_weights` 只保留 Top 10；`recent_clicked_answers` 只保留最近 10 条；`recent_queries` 只保留最近 5 条。
- 事件边界：只落盘 `search_query`、`recommendation_click`、`search_result_click`。
- 信号强弱：`search_query < recommendation_click < search_result_click`。
- 画像更新：同步直写画像，同时保留事件表；`search_result_click` 对 `topic_weights` 和用户向量的影响都强于 `recommendation_click`。
- 推荐点击更新：直接依据被点击 answer 自带的 topics 更新 `topic_weights`。
- 搜索点击更新：组合使用 query 命中 topics 与被点击 answer 的 topics 更新 `topic_weights`，其中重合部分视为更强确认。
- 交集处理：只规定“重合部分加得更多”，具体倍数留给配置与调优，不在 brief 中写死。
- `topic_weights` 更新：采用“加法累积 + 全局衰减 + Top 10 截断”。
- 冷启动：默认画像与个性化画像线性混合，混合权重由三类核心事件构成的 `behavior_score` 驱动。
- Topic 空间：保留 ZhihuRec 原生 topic，但只做轻清洗 / 轻合并，并选取一个可控子集。
- 数据规模：只重建一个小而精的内容世界，不追求大规模在线数据集。
- 用户范围：只围绕一个演示用户跑完整闭环，不做登录和多用户系统。
- 前端形态：轻前端 + 适度模块化，单页不是硬约束；职责是服务后端开发、调试、演示，而不是追求易用性。
- Answer 卡片内容：`V1` 只保留标题、摘要、author、topics 四类核心展示信息。
- Feed 调试字段：至少显示召回来源、基础召回分、topic 匹配分、query recall boost、最终分、fallback 标记、被选中原因。
- 被选中原因：采用规则模板生成，依据最高贡献项或主要贡献组合输出简洁解释。
- 后端形态：逻辑单体 + 模块分层，至少分出 `recommendation / search / profile / event`，不引入消息队列和服务注册。
- API 边界：至少提供 `/feed`、`/search`，以及拆开的点击事件接口；`/search` 一次请求内自然记录 `search_query` / `recent_queries` 并返回结果。
- 用户身份：接口保留 `user_id` 参数，但 `V1` 不做登录、JWT 和完整鉴权，只围绕默认演示用户运行。
- 基础设施：`V1` 先只用 MySQL；Redis 等第一版跑通后再加。
- 运行方式：本地单机跑通，必须提供一键初始化脚本。
- 重置能力：提供开发脚本形式的一键重置能力，不做前端入口。

### 参数与验证

- 参数原则：先定义相对关系，不在 brief 中写死所有公式与倍数；关键参数做成可调。
- 调优方式：优先采用离线回放测试，而不是只靠人工观察。
- 测试场景：至少覆盖推荐点击、搜索不点击、搜索点击三类场景。
- 主优化目标：响应性与稳定性的平衡。
- 关键次指标：`Search Carryover Gain@K`，用于支撑“搜索反哺推荐”的核心故事。
- 面试口径：参数不是拍脑袋定的，而是通过回放测试、代理指标和工程观察收敛得到的。
