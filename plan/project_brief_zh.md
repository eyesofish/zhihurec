# 项目简报：基于 ZhihuRec 的搜索-推荐闭环系统

## 0. 核心目标

这个项目**不只是**一门课的作业，也不是一个玩具级的推荐系统 Demo。

真正的目标是打造**一个核心项目，产出四份成果**：

1. 一份**可视化数据分析报告**
2. 一份**HCI 报告**
3. 一个看起来像真实搜索/推荐系统的**代码项目**
4. 一个由以上三者沉淀出的、可写进简历的**端到端项目故事**

对于 `V1`，首要目标应明确为：

> 一个面向简历过筛与面试讲述的工程项目

课程交付仍然重要，但应服务于这条主线，而不是压过工程项目本身。

这个项目应该帮助你：

- 通过这学期的两门课程
- 产出一个足够扎实、适合面试讲述的搜索/推荐系统项目
- 支撑用一条简历 bullet 或一个面试 hook 讲清完整系统链路与 trade-off
- 支撑搜索、广告、推荐、排序、召回、平台基础设施等方向的实习 / 校招面试

---

## 1. 项目定位

### 主轴

这个项目的主要定位应该是：

> 一个真实感较强的内容社区搜索 + 推荐闭环系统

在 `V1` 阶段，这个定位应进一步收敛为：

> 一个以工程闭环和面试可讲述性为优先目标的搜索 / 推荐项目

### 学术 / 讲故事切入点

这个项目的故事切入点是：

> 当用户从推荐流切换到搜索时，往往意味着当前推荐尚未完全捕捉用户此刻的真实意图。
> 因此，搜索应该被视为一种高意图信号，并反哺推荐系统。

需要明确区分两层含义：

- 这条叙事是项目在面试中的**故事 hook / 谈资**
- 它不是在实现层面取代推荐主链路成为新的工程主体

更直白地说：

> `A` 是工程实现主线，`B` 是面试表达主 hook

也就是：

- 工程上，项目仍以推荐系统主链路为主体
- 讲述上，最有记忆点的部分是“搜索如何作为高意图信号反馈到后续推荐”

这个切入点主要用于：

- HCI 叙事框架
- 数据分析叙事框架
- 面试中的项目讲述

### Progressive Logic of the Project

为了避免这个项目在表达上停留于“search 比 click 更重要”，这里需要把递进逻辑写清楚：

> conceptually large, operationally small

也就是：

> 叙事可以开大，但第一版工程改动必须做小、做稳、做得可验证

Our project follows a progressive logic from problem framing to minimal implementation.

#### Step 1: Problem Framing

We treat feed-to-search transition as a potential signal that the current recommendation state fails to satisfy the user's immediate intent.

换句话说：

- 用户从刷推荐转向主动搜索，不只是说明兴趣更强
- 也可能说明当前推荐流没有满足他此刻的具体需求

#### Step 2: Mechanism Hypothesis

This transition is modeled as a possible mode shift from passive consumption to active intent resolution.

也就是说：

- 我们不只是把 search 当作比 click 更强的信号
- 而是把这种行为看作用户状态的一次切换
- 用户可能从“随便刷刷”的被动消费模式，切到“我要解决一个具体问题”的主动意图模式

#### Step 3: System Implication

In principle, such a transition should affect both retrieval seed policy and user representation, since the system may need to move away from a recommendation state dominated by long-term preference.

这一步的含义是：

- 从系统设计上说，这种状态切换最终不应只影响一个权重
- 它原则上应同时影响召回 seed 的选择方式，以及用户表征的解释方式
- 因为系统可能需要暂时偏离“长期兴趣主导”的推荐状态，转而响应用户的即时任务

#### Step 4: Minimal First-Step Implementation

For the first implementation, we do not fully reconstruct the user model.
Instead, we only make a minimal intervention in retrieval seed selection:

- downweight shallow click-based seeds
- upweight search-aligned seeds

这就是第一版最小落地点：

- 降低浅层 click item 作为召回 seed 的权重
- 提高与近期 search 更对齐的 item 作为召回 seed 的权重

We start from retrieval seed selection because it is the smallest controllable intervention point.
It allows us to test whether feed-to-search transition carries useful corrective information,
without immediately introducing the complexity of rebuilding the entire user representation.

这一步的重要性在于：

- 它解释了为什么项目叙事可以讨论更大的系统问题
- 但第一刀实现只改一个很小的、可控的、可测试的机制点
- 这样项目既不像只是在调权重，也不会因为过早重构整个 user model 而显得空泛

因此，这一段应作为项目的核心递进逻辑：

- 先定义问题
- 再提出机制假设
- 再说明系统层面的推论
- 最后落到一个最小、可验证、可工程化的第一步实现

### 工程优先级

这个项目在工程上应优先被构造成：

- 一个真实的推荐流水线
- 然后是搜索对推荐的反馈
- 最后是事件级的闭环

这可以理解为：

- 推荐主链路决定项目是否站得住
- 搜索反馈机制决定项目是否足够有记忆点、足够好讲

这里的“真实”指的是：

- 有离线数据处理
- 有在线服务接口
- 有数据库持久化
- 有事件写回
- 有一个足够调试和演示用的极简前端

### V1 后端服务形态

`V1` 的后端应采用：

> 逻辑单体 + 模块分层

也就是说：

- 整体上仍是一个后端服务
- 但代码结构上应清晰划分 `recommendation`、`search`、`profile`、`event` 等模块
- 目标是既保持第一版落地简单，又保留足够强的工程叙事

在这个边界下，`V1` **不要**一开始引入：

- 消息队列
- 服务注册 / 服务发现
- 微服务拆分
- 以削峰填谷为目标的复杂异步架构

原因很明确：

- 第一版重点不是吞吐治理和分布式复杂性
- 第一版重点是把搜索、推荐、画像、事件回写做成一个可运行、可解释的闭环
- 逻辑单体已经足够支撑当前阶段的工程实践和面试讲述

优先级顺序：

1. **推荐主流程**
2. **搜索反馈进入推荐**
3. **用户反馈闭环**

## 2. 业务场景

**不要**再沿用之前那个多语言词汇平台作为项目外壳。

应改用下面这个场景：

> 一个类似知乎的内容社区。用户浏览推荐流，当推荐不足以满足需求时会发起搜索；而搜索行为又会进一步反馈到之后的推荐中。

这个项目可以借鉴**黑马点评**的后端工程思路，但**不能**继续停留在本地生活 / 店铺点评项目上。

### 可借鉴黑马点评的部分

可以借鉴：

- 用户系统
- 内容 / Feed 服务风格
- Redis / MySQL 的工程习惯
- 交互日志的记录思路
- 后端骨架 / 模块化风格

**不要**把重点放在：

- 优惠券秒杀
- 地理位置附近搜索
- 本地商家订单
- 本地生活业务逻辑

这个项目是**内容分发系统**，不是本地商业系统。

## 3. 数据集策略

### 主数据集

使用 **ZhihuRec** 作为唯一的主数据集基础。

ZhihuRec 会被用于三个目的：

1. **研究层**

   - 可视化数据分析
   - HCI 行为分析
   - 推荐转搜索现象分析

2. **建模层**

   - 构建用户兴趣信号
   - 构建 query-topic 映射
   - 构建内容 / 回答表示
   - 支持召回 / 排序逻辑

3. **演示数据基础**
   - 为 Demo 重建一个规模较小但足够真实的内容世界
   - **不要**在前端直接暴露完整原始数据集
   - 而是抽样 / 重建一个更像产品环境的小型内容世界

### V1 数据规模边界

`V1` 应明确采用：

> 小而精的数据规模策略

也就是说：

- 在线演示内容世界不追求大规模
- 优先保证内容实体、topic 关系、召回链路、画像更新和闭环逻辑都足够清楚
- 优先保证数据可调试、可查看、可解释

这个项目在第一版中不需要通过“很大的数据量”来证明价值。
第一版真正要证明的是：

- 你能把数据处理、召回、排序、事件回写、画像更新串成完整系统
- 你能解释系统为什么这样设计
- 你能把 trade-off 讲清楚

因此，数据规模应服务于工程闭环，而不是反过来让规模绑架实现复杂度。

### 重要原则

ZhihuRec **不只是**一个前端内容来源。

它应该扮演：

- 真实行为模式的来源
- 内容实体及其关系的来源
- 真实日志与 Topic 结构的来源

## 4. 核心产品结构

### 推荐主实体

主要的推荐实体应该是：

> **Answer**

### 辅助实体

- **Question**：回答的父级上下文
- **Topic**：连接 query、内容与用户兴趣的中间语义层
- **Author**：内容生产者
- **Query**：用户主动搜索时表达的意图
- **User**：拥有画像与行为历史的内容消费方

### 为什么以 Answer 作为主要推荐实体

原因是：

- 更接近用户真实消费对象
- 能支持更细粒度的推荐与反馈建模
- 更适合 Feed 卡片式展示
- 更容易调试和解释

### 产品展示策略

`V1` 前端在展示形态上应采用：

> 控制台式调试前端

其核心区域仍应以单列的 Answer 卡片流为主，但是否单页不是关键，关键是最利于后端调试与链路观察。

**不要**在第一版就把产品界面做得太精致。
前端首先应被视作：

- 调试控制台
- 推荐可观测性面板
- 闭环可视化工具

---

## 5. 前端理念

### 重要原则

这是一个由单个开发者完成的练习项目。
前端**不需要**为了真实用户而做得非常精致。

对 `V1` 来说，前端应明确采用：

> 能简则简

它的职责不是承担产品体验，而是最低成本地支撑后端开发、调试、演示和面试讲述。

更明确地说：

- 前端的首要作用是服务后端开发
- 前端不是为了真实用户易用性而设计
- 只要能把接口、召回结果、画像变化和事件闭环展示清楚，就已经达标

前端应该优先满足：

- 最低开发成本
- 最容易调试
- 最清晰地展示推荐决策
- 足够支撑面试时把系统链路讲完整
- 足够方便后端开发时直接观察系统行为

### 因此

应尽可能在前端直接暴露有用的调试信息。

每张 Answer 卡片可以展示：

- 问题标题
- 回答摘要
- 作者
- Topics
- 召回来源
- 召回分数
- 最终分数
- 是否触发 fallback
- 被选中的原因
- 匹配到的最近 query
- 用户兴趣匹配情况

在内容展示层面，`V1` 的 Answer 卡片应明确收敛为：

> 标题 + 摘要 + author + topics

也就是说：

- 不把第一版卡片继续扩展成更重的详情型组件
- 先保证内容信息足够支撑调试、解释和演示
- 详情展开能力不是 `V1` 的前提

在这些字段中，`V1` 的推荐 Feed 调试信息应至少强制展示：

- 召回来源
- 基础召回分
- `topic` 匹配分
- `query recall boost`
- 最终分数
- `fallback` 标记
- 被选中原因

其他字段可以作为补充展示，但这组字段应视为 `V1` 的必显集合。

其中，`被选中原因` 在 `V1` 中应明确采用：

> 规则模板生成

也就是说：

- 不做复杂自然语言生成
- 也不只是机械展示命中标签
- 应根据最高贡献项或主要贡献组合，生成一句简洁、可解释的原因说明

例如可以是：

- 因长期兴趣 / 搜索相关性匹配较高而进入前列
- 因 `topic` 匹配与最近搜索 boost 被提升
- 因主召回不足而由 `hot_or_fresh` 补齐

### V1 页面形态

`V1` 应采用：

> 轻前端 + 适度模块化

也就是说：

- 不追求复杂前端架构
- 推荐 Feed、搜索输入、搜索结果、用户画像摘要、调试信息可以按最利于调试的方式组织
- 可以是单页，也可以是少量简单页面；是否单页不是硬约束
- 更重要的是把 feed、search、profile、debug panel 拆成清晰模块
- 页面布局服务于开发调试效率，而不是服务于用户操作路径优化

### 推荐的首页设计

一个偏调试风格的 Feed 页面，展示：

- 当前用户画像摘要
- 带调试字段的推荐卡片
- 搜索输入与结果区域
- 点击等最小必要交互
- 可选的详情查看入口

这个前端本质上是：

> 一个服务后端开发的推荐系统控制面板，而不是一个打磨精致、追求易用性的产品 UI

## 6. 用户画像设计

### 重要澄清

用户画像**不等于**用户 embedding 向量。

用户画像应理解为：

> 系统为一个用户维护的全部结构化信号

embedding 向量只是画像中的一种紧凑表示。

### V1 的用户画像应包含

- `topic_weights`
- `recent_clicked_answers`
- `recent_queries`

这个选择很重要，因为它支持：

- 可解释性
- 调试
- 面试中的项目讲述
- 后续搜索反馈整合

### 为什么采用这种画像结构

- `topic_weights` = 中期、可解释的兴趣状态
- `recent_clicked_answers` = 最近、具体的消费行为
- `recent_queries` = 最近的高意图但相对更弱的搜索意图信号

对于 v1 来说这已经够了。
**不要**把画像设计得过度复杂。

其中，`topic_weights` 在 `V1` 中应明确为：

- 只保留权重最高的 Top `10` topics
- 主要服务于兴趣解释、召回增强和前端调试展示
- 不在第一版中维护过大的 topic 权重集合

同时，`topic_weights` 的更新策略应明确采用：

> 加法累积 + 全局衰减

也就是说：

- 每次发生会影响画像的点击事件时，先对现有 `topic_weights` 做一次统一衰减
- 再根据本次事件命中的 topics 做加分累积
- 最后只保留 Top `10` topics

这样做的原因是：

- 保留兴趣随行为累积的直觉
- 避免很早的历史行为永久主导画像
- 让画像能随着最近行为逐步漂移
- 在第一版里仍然保持实现简单、可解释

对于 `recommendation_click`，`V1` 应进一步明确为：

> 直接依据被点击 answer 自带的 topics 更新 `topic_weights`

也就是说：

- 不再额外引入 question topics 补充逻辑
- 不做二次 topic 映射再更新
- 第一版优先保持更新路径短、清晰、可解释

对于 `search_result_click`，`V1` 应进一步明确为：

> 组合使用 `query -> topic` 命中的 topics 与被点击 answer 自带的 topics 更新 `topic_weights`

也就是说：

- 不是只看 query topics
- 也不是只看 answer topics
- 两者组合后的重合部分，应被视为更强确认信号

这里同样应保持一个明确边界：

> 只规定“重合部分加得更多”，不在 `V1` brief 中写死具体倍数或公式

也就是说：

- 交集 topics 的更新强于非交集 topics
- 但精确权重关系留给实现配置与测试调优
- 不在第一版文档中提前锁死 `2x`、`3x` 或更复杂规则

这样做的原因是：

- query 暴露了用户主动意图
- answer click 完成了内容层面的偏好确认
- 两者合起来更符合“搜索先表达意图、点击再确认意图”的闭环逻辑

其中，`recent_clicked_answers` 在 `V1` 中应明确为：

- 只保留最近 `10` 条点击记录
- 每条记录保留最小必要时间信息
- 主要服务于用户向量构造与近期行为解释

其中，`recent_queries` 在 `V1` 中应明确为：

- 只保留最近 `5` 条 query
- 主要服务于轻量意图记录与 topic-aware boost
- 不把第一版做成完整的搜索历史管理系统

### V1 画像存储形态

`V1` 的用户画像应采用：

> 一张用户画像表 + 偏结构化字段存储

也就是说：

- 不把画像拆成很多张明细表
- 优先使用一张 `user_profile` 或等价主表承载核心画像
- `topic_weights`、`recent_clicked_answers`、`recent_queries` 可以使用 JSON、文本或其他便于快速落地的结构化字段存储

这样做的原因是：

- 第一版实现最简单
- 最容易同步直写更新
- 最容易调试和直接查看画像内容
- 最符合当前“闭环优先、复杂度靠后”的项目边界

后续如果画像规模、更新频率或查询模式变复杂，再考虑拆分成更细的画像表或明细表。

## 7. 冷启动 / 默认画像

由于系统从 ZhihuRec 起步、且没有真实在线用户，v1 应使用：

> **默认冷启动画像**

这比把它叫做“平均用户画像”更合适。

### 目的

它应当充当：

- 全局先验
- 新用户的起点
- 当用户特定信号稀疏时的一层轻量基础

### v1 冷启动策略

使用：

- 全局 topic 分布
- 基于 topic 的热门内容

之后可以再和用户特定信号进行混合。

`V1` 的默认画像与个性化画像应明确采用：

> 线性混合

也就是说：

- 在用户行为很少时，默认冷启动画像占更高权重
- 随着用户点击、搜索结果点击等有效行为增多，个性化画像权重逐步上升
- 默认画像权重逐步下降，而不是在某一个时刻被硬切掉

这样做的原因是：

- 比“有了一个行为就完全切到个性化”更稳
- 更符合冷启动到个性化过渡的直觉
- 更容易在面试中解释默认先验如何逐步退场

用于驱动这条线性混合的，不应只是单一事件数，而应是：

> 综合行为分数 `behavior_score`

在 `V1` 中，`behavior_score` 只由三类核心事件组成：

- `search_query`
- `recommendation_click`
- `search_result_click`

并且三者权重应满足：

- `search_query` = 低权重信号
- `recommendation_click` = 中权重信号
- `search_result_click` = 高权重信号

也就是说：

- `search_query` 可以直接进入冷启动混合权重的计算
- 但它只作为较弱的意图信号参与
- 真正更快推动系统从默认画像转向个性化画像的，仍然是点击类行为，尤其是 `search_result_click`

对于 `behavior_score -> alpha` 这一步，`V1` 应进一步明确为：

> 只保留相对趋势，不在 brief 中写死公式

也就是说：

- 只需要明确“行为越多、行为越强，个性化权重越高”
- 不在 `V1` brief 中提前锁死精确函数形式
- 具体计算方式在实现中应保留为可调参数或可替换逻辑

同时需要明确一个很重要的工程与面试原则：

> 参数不是拍脑袋定的，而是通过测试和调优逐步得到的

这意味着：

- `behavior_score` 的各事件权重应支持实验和调参
- 默认画像与个性化画像的混合参数应支持测试验证
- 最终保留下来的参数，应能在面试中解释为“基于测试结果和工程观察收敛得到”，而不是随意指定

`V1` 在参数调优验证上应优先采用：

> 离线回放测试

也就是说：

- 选取固定的单用户行为序列进行回放
- 观察不同参数下画像、召回结果和排序结果如何变化
- 用这套回放结果支撑参数选择与工程调优

`V1` 的离线回放测试场景，至少应覆盖：

- 推荐点击场景
- 搜索但不点击场景
- 搜索并点击结果场景

这样设计的原因是：

- 能同时覆盖中等信号、弱信号和强信号
- 能验证为什么 `search_query` 不能直接强更新画像
- 能验证为什么 `search_result_click` 应高于 `recommendation_click`

这样做的原因是：

- 最符合当前单用户闭环项目的形态
- 最容易验证“行为变化是否真的驱动了推荐变化”
- 最容易在面试中说明参数设计依据来自测试而不是直觉

这类参数调优在 `V1` 中的核心优化目标，应明确为：

> 响应性与稳定性的平衡

也就是说：

- 一次高意图行为之后，下一轮推荐应明显朝新意图方向变化
- 但系统又不能因为一次行为就发生过度漂移

对于“用户是否更满意”这件事，`V1` 应保持严谨表达：

> 不直接宣称已经在线证明用户满意度提升，而是通过满意度代理指标证明推荐结果更贴近用户当前意图

更适合 `V1` 的量化代理指标可以包括：

- `Intent Responsiveness@K`
  含义：高意图行为发生后，下一轮 Top-K 推荐中，与目标 topic 对齐的内容占比是否明显上升
- `Search Carryover Gain@K`
  含义：`search_result_click` 之后，下一轮推荐是否更快出现与该搜索 topic 相关的内容
- `Overreaction Penalty`
  含义：一次事件后，推荐列表是否发生过度漂移；如果变化过猛，则该项变差
- `Top-K Stability`
  含义：在没有新的强信号时，相邻两轮推荐结果应保持一定稳定性，而不是无意义抖动

如果需要更工程化地补充，还可以记录：

- `Fallback Exposure Rate`
  含义：推荐结果里 fallback 内容占比是否过高；这个指标更偏系统健康度，而不是直接满意度

在这些指标中，`V1` 的指标口径应进一步明确为：

> 主指标采用“响应性 + 稳定性”的组合视角

也就是说：

- 不把单一指标当成唯一目标
- 参数选择的主判断标准，是系统是否在“足够响应新意图”和“不过度漂移”之间取得平衡

同时还应明确一个关键次指标：

> `Search Carryover Gain@K` 是服务项目故事线的重点指标

原因是：

- 它最直接对应“搜索反哺推荐”这条核心 hook
- 它最容易向面试官解释：搜索点击之后，后续推荐到底有没有真正承接搜索意图
- 它应作为主指标之外、最重要的故事型辅助指标存在

因此，面试中的表述最好是：

> 我们通过离线回放测试对参数做 sweep，主看响应性与稳定性的平衡，同时重点观察 `Search Carryover Gain@K`，用它来支撑“搜索意图是否真正传导到了后续推荐”这条核心故事，而不是拍脑袋选参数。

这条规则的目的，是让系统既能感知“用户已经开始主动表达意图”，又不会因为一条 query 就过快抛弃默认先验。

重要的是：
不要过度设计冷启动。
它只需要回答两个问题：

1. 当用户没有个人历史时，系统展示什么
2. 系统如何从默认画像逐步过渡到个性化画像

### V1 用户范围边界

`V1` 应明确采用：

> 单用户闭环演示

也就是说：

- 系统围绕一个默认用户或演示用户跑完整闭环
- 不做完整登录系统
- 不做多用户切换和多用户对比展示

这样做的原因是：

- 最容易把“这次行为如何改变下一轮推荐”讲清楚
- 最容易调试画像更新、事件回写和结果变化
- 最符合当前“第一版尽量简单，但闭环必须成立”的目标

如果后续需要增强演示性，再考虑加入“重置画像”或少量预设用户，但这不是 `V1` 的前提。

## 8. 推荐架构（v1）

### 推荐亮点

这个核心项目的亮点应该是：

> 具备多路轻召回 + 状态感知重排 + fallback 的推荐流水线

这里的重点不应再定义为“先把完整双塔做出来”，而应定义为：

> 先用一个结构正确、解释清楚、工程复杂度可控的推荐系统，把 feed-to-search transition 这条研究主线真正跑起来

### 为什么不先做大而全推荐系统

基于当前项目边界，更稳的判断是：

> 当前数据条件与项目目标，并不支持先做一个完整、重训练、端到端的工业级召回系统

相比之下，一个：

> 多路轻召回 + 状态感知重排

会更符合 `V1` 的工程约束。原因是：

- 能体现推荐系统最基本的结构：召回、重排、反馈闭环
- 能把研究重点集中在 feed 转 search 的用户状态切换上
- 不会因为过早做重召回建模而稀释项目的真正亮点
- 更适合做成一个轻量、可讲清楚、可验证的工程 demo

换句话说，`V1` 的目标不是：

> 先证明我能做一个很重的召回系统

而是：

> 先证明我能在推荐失配场景下，用一个简单但合理的系统结构接住用户的短期意图变化

### V1 三层推荐骨架

`V1` 更适合明确采用下面这套三层方案：

#### 第 1 层：多路轻召回

先召回一个“不太蠢”的候选集，而不是一开始追求最优全局召回。

推荐的最小召回路由是：

1. 热门 / 新鲜内容
2. 用户长期兴趣 topic
3. 最近 search query 相关的 topic 或内容
4. 少量 exploration 内容

这一层的目标不是“排最对”，而是：

> 尽量别漏掉可能对的东西

也就是说，这一层首先解决的是候选覆盖问题，而不是精细排序问题。

#### 第 2 层：轻量重排器

这才是 `V1` 的核心灵魂。

重排器不需要复杂模型，一个简单、可解释的加权打分公式就够了。
输入特征应优先采用几类廉价、稳定、可调试的信号：

- 用户长期偏好
- 用户短期状态
- 当前 query / search 信号
- item 自身属性
- user-item / user-topic 匹配程度

可以直接写成：

> `score = a * long_term_match + b * short_term_match + c * query_match + d * quality + e * diversity + f * mode_switch_adjustment`

#### 第 3 层：状态开关

这就是项目真正的研究重点：

> feed-to-search transition

当用户表现出从被动刷到主动找的迹象时，排序权重发生切换：

- 长期偏好权重下降
- 当前 intent 相关权重上升

这一层在 `V1` 中不需要被做成一个独立大模型。
先把它实现为一个：

> gating score

就已经足够成立。

### 一个更稳的工程解释

这套方案的价值，不在于“已经做了完整双塔训练”，而在于它非常符合工业界第一版系统的落地顺序：

> 先把 system position 站稳，再逐步增加 representation complexity

也就是说：

- 第一版不急着追求端到端大模型或重训练双塔
- 而是先承认一个现实：只要 `item` 侧表示足够稳定，`user` 侧表示足够便宜且可更新，系统就已经可以跑出一个成立的个性化闭环
- 对 `V1` 来说，先完成“可上线、低延迟、可回退、可扩展”的轻量推荐链路，比先完成一个复杂训练流程更重要

这类思路与 DoorDash 早期个性化搜索 / 推荐系统的工程取向是一致的：

- 重活尽量前置到离线
- 在线阶段只做轻计算
- 先把个性化能力嵌入现有系统，而不是一开始就重构整个架构

对应到当前项目，可以明确写成：

- 离线阶段负责构建 item 的 topic、tag、可选 embedding、流行度、时效性和基础质量分
- 在线阶段只读取用户画像与最近状态、生成少量候选、执行轻量重排
- 一旦个性化状态不可用，应能够自然回退到热门 / 默认推荐逻辑

这里最关键的不是“模型先进”，而是：

> 先把个性化闭环做成一个不会先把系统搞死的版本

因此，`V1` 不应把“是否完整训练了双塔”当成成败标准，而应把下面这些作为更重要的判断标准：

- 是否真的有离线 item 表示产出
- 是否真的有在线 user 状态聚合
- 是否真的有稳定的多路候选生成链路
- 是否真的有失败可回退的默认路径
- 是否真的能把搜索反馈、画像更新和后续推荐结果串成可验证闭环

### V1 item 侧离线产物

`V1` 的 item 侧离线准备应优先包括：

- `Answer` 的 topic / tag
- 可选的内容 embedding
- popularity / freshness / baseline quality

如果要做 embedding，最合适的边界仍然是：

> `question_title + answer_summary + topics`

这样做的原因是：

- 比只用标题或摘要更完整
- 比直接引入全文更轻、更稳
- topic 信息能帮助表示层与解释层保持一致

这意味着第一版**不需要**：

- 引入完整 answer 全文做主向量
- 维护多套长短文本向量
- 为内容表示设计过重的字段拼接策略

这一策略的好处是：

- 最符合当前项目“训练不重、工程链路要真”的边界
- 能把 item 表示、用户状态、搜索反馈三者自然串起来
- 面试里容易讲清系统链路和 trade-off

### V1 user 侧规则聚合

`V1` 的 user 侧不应先追求一个重 user tower，而应先做规则聚合。

最合适的最小集合包括：

- 长期兴趣分布：过去 `7` 天或 `30` 天点击过哪些 topic
- 短期兴趣分布：过去 `1` 天或当前 session 的 topic
- 搜索活跃度：最近 search 次数、search session 占比
- feed 失配迹象：最近 CTR 下降、连续 skip、dwell 下降

这就是“用户向量规则聚合”在当前项目里真正应落地的含义：

- 先做简单统计量
- 先做可解释状态变量
- 先不训练复杂 user representation

如果后面确实需要保留向量化 user 表示，也应放在可选增强位，而不是 `V1` 主体。

### 面试表述边界

对于这套实现，最稳妥的表达不是：

> 先做了一个完整双塔推荐系统

而更准确的表达应是：

> 实现了一个 `search-aware re-ranking for feed recommendation` 的轻量推荐系统：通过多路轻召回形成候选集，再利用用户长期偏好、短期状态、当前 query 信号与 mode-switch score 对候选进行状态感知重排

这样表述的好处是：

- 不会夸大召回建模部分
- 能准确体现这个项目真正的研究重点
- 与当前 `V1` 的实现边界和演示目标一致

### v1 的召回桶

一开始应使用下面四类轻召回桶：

1. `hot_or_fresh`
2. `long_term_topic`
3. `recent_search_topic`
4. `exploration`

原因：

- 最小化但足够真实的闭环
- 容易解释
- 容易调试
- 更贴合当前研究问题
- 能覆盖冷启动、意图切换和探索场景

在候选规模上，`V1` 应进一步明确为：

> 多路轻召回先产生几十到几百个候选，再压到最终 Top `10`

这样做的原因是：

- 足够体现“召回”和“排序”是两层不同作用
- 第一版实现仍然足够轻
- 面试里也容易讲清候选集是如何被压缩的

其中，`hot_or_fresh` 在 `V1` 中应明确为：

> 全局热门 / 新鲜 answer

也就是说：

- 既可承担默认兜底
- 也可承担“别让候选集太蠢”的基础覆盖
- 第一版不需要再做复杂多级 fallback

这里的 `exploration` 应明确为：

> 故意混入少量与当前主兴趣不同、但仍合理的主题内容

也就是说：

- 它不是随机噪声
- 它是为了给系统保留发现新兴趣和避免主题过窄的空间
- 它也让推荐结果更像真实产品，而不是只会围绕单一 topic 打转

`V1` 的 fallback 触发条件也应明确为：

> 只要主召回数量不足 Top `10`，就用 `hot_or_fresh` 补齐

也就是说：

- 不额外引入分数阈值判断
- 不要求主召回完全为空才触发
- 第一版只按数量是否补满来决定是否兜底

### v1 不要一开始加太多召回桶

**不要**从下面这些开始：

- 单独的 follow-based bucket
- 多阶段复杂 rerank 树
- 完整双塔 + 大规模 ANN 基础设施
- 重训练 user tower / intent encoder

这些都可以后面再做。

### 为什么

第一目标是：

- 先让推荐主循环真实跑起来
- 先让 fallback 可见
- 先让系统行为可解释
- 先把多路轻召回与状态感知重排跑通，再考虑更重的召回建模

## 9. 搜索反馈设计

### 主切入点

搜索应被视为：

> 用户具有更主动、更具体需求的信号，而这种需求可能尚未被推荐流充分满足

这一点在项目里的定位应明确为：

- 它是最核心的故事点和面试谈资
- 但它建立在推荐主链路已经成立的前提上
- 它是推荐系统的增强闭环，不是替代推荐系统本身

### 重要细节

**不要**过度宣称“搜索一定意味着不满意”。
更稳妥的解释是：

> 搜索是一种更高意图的信号，它经常反映尚未被满足、或尚未被系统捕捉到的需求

### V1 搜索模块边界

`V1` 的搜索模块应定义为：

> 一个极简但真实的搜索链路

它不是一个独立打磨的“大而全搜索产品”，而是在尽量简单的前提下，同时满足三件事：

1. 搜索侧的**召回机制本身能够成为亮点**
2. 能完整支撑“搜索反哺推荐”的故事
3. 能真实改动用户画像

因此，`V1` 搜索至少要具备：

- 一个真实可用的搜索输入入口
- 一个最小可用的搜索结果列表
- 一条可解释的搜索召回链路
- 搜索点击后的事件回写
- 搜索行为对后续推荐的实际影响

对于“输入 query”这件事，`V1` 应进一步明确为：

> 不在输入 UI 上过度投入，优先提供方便后端模拟真实输入的开发方式

也就是说：

- 不追求产品化的搜索输入体验
- 前端输入框只需要满足最小可用
- 更重要的是提供方便后端调试和回放的脚本化输入方式

`V1` 在这部分应优先提供：

> 命令行脚本，既支持单个 query 调试，也支持 query 序列回放

这样做的原因是：

- 同一套入口可以同时服务后端调试与离线回放
- 比单纯依赖前端手工输入更高效
- 更适合单用户闭环项目的工程验证方式

其中，`query` 序列回放的输入格式应优先采用：

> JSON 序列

并且应支持携带：

- 时间信息
- 动作类型

这样做的原因是：

- 后续可以自然扩展到 `search_query`、`search_result_click`、`recommendation_click` 等不同事件
- 比纯文本一行一个 query 更适合回放真实行为序列
- 更方便后续测试脚本、参数调优和事件重放复用同一套输入结构

`V1` 在动作粒度上，应至少支持：

- `search_query`
- `search_result_click`
- `recommendation_click`

这样正好与当前 `V1` 已经定义好的三类核心事件保持一致。

这类方式可以包括：

- 开发脚本触发搜索请求
- 固定 query 序列回放
- 便于模拟真实输入流的调试入口

核心目标不是把“输入框”做精致，而是让你能低成本模拟真实 query 输入并观察系统反应

### V1 搜索结果来源策略

`V1` 的搜索结果来源应采用：

> `query -> topic -> answer` 为主，`hot_or_fresh` 只用于补齐不足结果

也就是说：

- 搜索主结果应来自 query-topic-answer 这条相关性链路
- 热门内容不是搜索主排序逻辑，只在结果不足时作为兜底补齐
- 默认不采用“similarity 分数 + 热门分数直接统一加权混排”作为 `V1` 主策略

更具体地说，`V1` 更适合：

- 先按相关性链路召回并排序搜索候选
- 如果相关候选不足，再从 `hot_or_fresh` 补充剩余结果位
- 在结果和调试信息中明确标识每条内容的来源

这样设计的原因是：

- 更符合搜索场景下“相关性优先”的直觉
- 更容易保证“搜索反哺推荐”的故事不被热门逻辑冲淡
- 更容易调试和解释为什么某条结果会出现
- 更适合在面试中讲成清晰的 bucket + fallback 机制

如果后面确实需要在少量结果位中引入热门信号，也应优先采用：

- 分桶后补齐
- 或者对热门结果施加强 penalty / 强配额限制

而不是在第一版里直接把相关性和热门做成等价的混合加权。

这里的“可解释搜索召回链路”应尽量体现为：

> query -> topic -> answer

也就是说，搜索的亮点不在于复杂排序，而在于：

- query 如何被映射到 topic
- topic 如何召回到 answer
- 这些信号如何再反馈回推荐系统

为了保持 `V1` 简洁，搜索模块**不要**一开始加入：

- 复杂筛选器
- 多阶段搜索排序
- Query Rewrite / 扩写纠错等复杂查询理解
- 历史搜索、联想词、个性化搜索页等产品化细节

### 信号强弱层级

将用户信号分层看待：

#### V1 最小事件集中的弱信号

- 原始搜索 query

#### V1 最小事件集中的中等信号

- 推荐点击

#### V1 最小事件集中的强信号

- 搜索结果点击

如果从完整系统角度看，未来仍可以继续保留下面这类更强信号作为扩展：

- like
- favorite
- 对相似内容的重复点击

对 `V1` 来说，这里的关键不是把所有事件都做全，而是把三类核心信号的相对强弱讲清楚：

- `search_query` = 弱信号
- `recommendation_click` = 中等信号
- `search_result_click` = 强信号

这样定义的原因是：

- `search_query` 只表达意图，不代表偏好确认
- `recommendation_click` 代表用户对被动曝光内容产生兴趣，但意图强度仍弱于主动搜索后的点击
- `search_result_click` 发生在用户主动表达 query 之后，更适合作为强确认信号
- 因此，`search_result_click` 在画像更新和用户表示构造中的权重都应高于 `recommendation_click`

这里的要求首先是：

> 保证相对强弱顺序，而不是在 `V1` brief 中写死具体倍数

也就是说：

- 只需要明确 `search_result_click_weight > recommendation_click_weight`
- 具体数值在实现中应作为可配置参数保留
- `V1` 文档不预先锁死成 `2x` 或 `3x`

### v1 的搜索更新规则

使用下面这条规则：

- 当用户输入搜索 query 时：

  - 写入 `recent_queries`
  - **不要**直接更新长期 `topic_weights`

- 当用户点击某个搜索结果时：
  - 再更新 `topic_weights`

在 `V1` 中，更具体的更新顺序应是：

1. 先对现有 `topic_weights` 做全局衰减
2. 再按本次事件对应的 topics 做加法累积
3. 最后截断到 Top `10`

这是一个有意做出的权衡：

- 原始 query = 更轻、更噪的意图信号
- 推荐点击 = 中等强度的兴趣确认
- 搜索点击 = 更强的偏好确认

在 `V1` 中，这里的“更强”应首先体现为：

- `search_result_click` 的更新权重高于 `recommendation_click`
- 具体倍数不在 brief 中写死，而留给实现阶段配置

这点应在项目说明里明确写清楚。

## 10. Query 到 Topic 的映射

### query-topic mapping 在做什么

它做的事情是把：

> 用户临时性的表述方式 -> 系统内部稳定的 topic 空间

换句话说：

- 用户通过 query 文本表达意图
- 系统需要一个稳定的内部语义层
- topic 充当搜索信号和推荐内容之间的桥梁

目标链路是：

> query -> topic -> answer

### v1 的要求

这种映射必须满足：

- 足够真实
- 低延迟
- 低成本
- 易于调试
- 易于在面试中解释

### V1 Topic 空间边界

`V1` 的 Topic 空间应采用：

> 保留 ZhihuRec 原生 topic 体系，但只做轻度清洗 / 合并，并只取一个可控子集

也就是说：

- 不在第一版重新设计一整套新的 topic taxonomy
- 先检查原生 topic 的数量、分布和可用性
- 再选择一个足够支撑闭环和面试讲述的 topic 子集
- 只做必要的轻清洗、轻合并、去噪和去异常 topic 处理

这个边界的核心目的是：

- 保留数据真实性
- 避免 topic 体系治理本身变成一个大工程
- 让 `query -> topic -> answer` 这条链路足够稳定、足够好讲
- 让第一版把重点放在闭环跑通，而不是放在 topic 设计本身

因此，topic 选择原则应优先考虑：

- 高频且语义相对稳定的 topic
- 能覆盖主要内容与 query 场景的 topic
- 数量可控、便于调试和观察的 topic

### 最终决策

使用：

> **基于 ZhihuRec 离线统计得到的 query-topic 共现关系**
>
> - 在线阶段通过倒排映射做轻量级 keyword / token 查找

v1 **不要**使用在线 embedding 相似度。

这里隐含的前提是：

- query-topic 映射只在选定的 topic 子集上构建
- topic 子集应尽量稳定、干净、可解释
- 如果原生 topic 过多或过散，先裁子集，再做映射统计

### 为什么不先上 embedding

因为 v1 的优先级是：

- 延迟
- 低工程成本
- 可解释性
- 可调试性
- 简历 / 面试表达清晰

基于 embedding 的语义匹配可以明确写成：

> 面向更强语义泛化能力的后续扩展

### 在线策略

运行时：

- 对 query 做归一化 / 分词
- 查 query-topic 映射
- 按配置文件中的动态阈值选取相关 topics，而不是写死固定 Top N
- 基于这些 topics 找到相关 answer 候选
- 优先返回相关性链路得到的搜索结果
- 如果相关候选不足，再用 `hot_or_fresh` 补齐
- 返回固定 Top `10` 的最小可用且可解释的搜索结果列表
- 将这些 topics 作为轻量级召回增强信号

如果前端暴露调试信息，搜索结果最好也能展示：

- query 命中的 topic
- 结果命中的 topic / answer
- 搜索结果来源或命中原因
- 每条结果的匹配分组成

`V1` 在搜索结果调试展示上，应至少满足：

> topics + 来源 + 匹配分组成

这样做的原因是：

- 能直接解释“为什么是这条结果被召回”
- 能把 query-topic-answer 这条链路讲清楚
- 能服务后端调试和面试演示

### V1 query-topic 选取边界

`V1` 中，`query -> topic` 的在线选取应采用：

> 动态阈值 + 配置文件驱动

也就是说：

- 不在 brief 中写死一次必须取 Top `1` / `3` / `5`
- 配置形式优先采用“分数阈值 + 最大 topic 数”
- 具体阈值规则放在配置文件中
- 实现时应支持根据 topic 分数分布动态决定保留多少个 topic

这样做的原因是：

- 比固定 Top N 更灵活
- 更符合后续参数调优与回放测试的需要
- 面试里可以说明“topic 选取规则是可配置、可验证的”，而不是硬编码常数

在当前 `V1` 边界下，这里的 `score` 也应优先理解为：

> 基于离线 `query-topic` 统计关系的匹配分数，而不是在线 embedding 余弦相似度

也就是说：

- 在线阶段优先走更快的倒排查找 / 共现匹配分数
- 不把 query-topic 映射做成在线 embedding 检索问题
- embedding 更适合保留在 answer 向量召回层，而不是优先放到 query-topic 这一层

更具体地说，`V1` 的 `query -> topic score` 应采用：

> 离线共现分数聚合 + 少量规则加成

也就是说：

- 每个 query token 或短语先从离线映射表中取出对应的 topic 候选与基础分数
- 在线阶段对这些基础分数做聚合
- 在此基础上允许加入少量规则加成

这些规则加成应保持克制，优先考虑：

- 完整短语命中高于单 token 命中
- 多 token 一致指向同一 topic 时可获得额外加成

`V1` 在这部分应明确收敛为：

> 只保留两条规则加成

也就是说：

- 保留“完整短语命中高于单 token 命中”
- 保留“多 token 一致指向同一 topic 时加成”
- 暂不继续扩展更多规则系统

这一层的设计目标是：

- 保留统计映射的主线
- 让 query-topic 匹配更像真实搜索工程
- 但仍然保持第一版的实现简单、可配置、可调试

这样做更符合当前项目的要求：

- 更快
- 更便宜
- 更容易调试
- 更容易解释
- 不违背“`V1` 不做在线 embedding query-topic 映射”的既有边界

### 存储策略

最终目标架构：

- MySQL 作为 source of truth
- Redis 作为缓存 / 预热层

但实现顺序应该是：

1. `V1` 先只用 **MySQL** 跑通
2. Redis 明确放到第一版跑通之后再加
3. 如有需要，再把 Redis 作为缓存 / 预热层接入

这是有意为之。
系统应优先保证：

- 先有功能闭环
- 再做性能优化

`V1` 在运行时存储边界上，应进一步明确采用：

> MySQL 作为运行时唯一真源

也就是说：

- 在线服务在运行时只从 MySQL 读取推荐、搜索、画像、事件和调试接口所需数据。
- 离线构建产物只负责初始化、导入、重建和本地检查，不作为在线服务直接读取的运行时数据源。
- 如果某类数据需要被在线服务消费，第一步应是把它导入 MySQL，而不是绕过数据库直接读派生文件。
- 这样可以保证运行时状态、调试观察和后续接口实现都围绕同一份真源展开。

更具体地说，`V1` 的边界是：

- 系统不能依赖 Redis 才能运行
- 系统不能依赖本地离线产物文件直接参与在线请求处理
- 不在第一版中处理缓存一致性、预热策略和失效策略
- Redis 只作为后续性能优化项保留

## 11. 推荐系统如何使用搜索信号

### 主流架构参考

更主流的工业推荐链路通常会区分：

- 召回：先决定哪些内容值得进入候选集
- 粗排 / 预排：快速压缩候选规模
- 精排：对更小候选集做更细打分

在这类架构里，`query` 这类高意图但弱确认信号，最自然的入口通常是：

> 先影响召回，再在系统更成熟时逐步进入粗排 / 精排特征

原因是：

- `query` 首先改变的是候选空间
- 它更适合表达“当前想看哪类内容”
- 它不应该在第一版里被过早做成复杂精排信号

### v1 设计

`recent_queries` 首先应该被用于：

> 通过 topic-aware 的过滤 / boosting 去影响召回

v1 **不要**把 recent queries 直接喂进双塔模型的用户塔。

更具体地说：

- `recent_queries` 不参与用户向量构造
- 它只通过 `query -> topic` 映射影响候选生成、召回增强和候选空间调整
- 真正进入用户向量主体的是 `recent_clicked_answers` 与 `topic_weights`

一开始也**不要**把搜索信号直接用于粗排 / 精排模型。

换句话说：

- `V1` 里，raw `search_query` 本身不直接进入精排打分
- 搜索信号优先影响召回层；如需进入排序，也应先被压缩成少量可解释状态特征
- 不把 raw `search_query` 直接做成精排特征
- 也不在第一版里单独搭建更重的粗排 / 精排双模型体系

### feed-to-search 不要先理解成召回革命

在 `V1` 中，更稳的理解方式是：

> 先把它看成一个排序态切换问题，而不是召回层革命

也就是说：

- 用户仍然大概率在同一个内容池附近活动
- 但他此刻不一定还想吃默认 feed 那套排序逻辑
- feed 到 search 的切换，首先说明的是“当前排序状态可能已经不对味”，而不一定要求立刻重做整个召回架构

因此，这个项目在第一版里更适合做的是：

- 保持主召回链路尽量稳定
- 把 feed-to-search transition 先拆成一组廉价、可解释、可实时更新的状态特征
- 再把这些特征作为排序 gating signal，去调节当前 intent 与长期偏好的相对权重

### feed-to-search 的廉价状态特征

不要一开始就引入抽象的 `search_mode` 二值标签。
更适合 `V1` 的方式，是先把“用户从刷推荐转去搜索”拆成几组可直接计算的便宜特征。

#### 1. 最近主动性上升

用于观察用户是否突然从被动消费切换到主动寻找：

- `recent_search_cnt_10m`
- `recent_search_cnt_1d`
- `search_after_feed_view_gap`
- `search_session_ratio`

这组特征表达的是：

> 用户开始主动出手了

#### 2. 最近推荐失配加重

用于区分“用户本来就爱 search”与“因为当前 feed 没接住，所以开始 search”：

- `feed_ctr_drop_vs_baseline`
- `feed_dwell_drop_vs_baseline`
- `skip_streak`
- `impression_without_click_cnt_recent`

这组特征表达的是：

> 用户不是单纯更活跃，而是当前推荐可能开始不对味

#### 3. search 与既有画像偏离

用于判断这次搜索是否偏离长期兴趣或最近曝光轨迹：

- `query_topic_vs_user_profile_sim`
- `query_topic_vs_recent_feed_topic_sim`
- `query_topic_novelty`
- `category_switch_flag`

这组特征特别适合当前项目的边界，因为它可以直接复用：

- 离线构建的 topic / 内容表示
- 在线维护的 `topic_weights`
- 最近 feed 暴露或点击记录

#### 4. search 表达强度

用于区分“随便搜一下”与“带着明确任务来”：

- `query_len`
- `query_has_specific_entity`
- `query_has_filter_term`
- `query_refine_from_prev_query`
- `query_repeat_with_more_specific_terms`

这组特征表达的是：

> 系统现在是否应该从泛兴趣推荐，切向更 task-oriented 的内容组织方式

#### 5. search 后反馈是否验证了当前判断

不要只看用户有没有 search，还要看 search 之后的行为是否证明这次 mode switch 真的成立：

- `search_result_click`
- `search_to_dwell`
- `search_to_save_or_like`
- `search_to_conversion`
- `search_reformulation_cnt`

如果 search 之后点击率、停留或后续互动明显变好，说明当前状态切换判断是有意义的。
如果 search 后仍不断改写 query，则说明系统连主动 intent 也没有接住。

### V1 的状态分数与 gating

这些特征在 `V1` 中不应直接导向一个重模型，而更适合先被压成一个轻量状态分数：

> `mode_switch_score = a * 主动性上升 + b * feed失配 + c * topic偏离 + d * query明确度`

这不是为了做一个学术上复杂的状态识别器，而是为了给当前排序层一个：

> 状态感知开关

最朴素的使用方式可以是：

- 分数低：继续走默认 feed 排序
- 分数中：提高当前 search topic 相关内容的权重
- 分数高：明显降低长期偏好权重，提高当前 query / topic 权重

这里的关键点是：

- 它影响的是现有排序器中的权重关系
- 它不是直接替换现有推荐系统
- 它也不是要求第一版就把 search 信号喂进完整双塔或粗排 / 精排模型

因此，`V1` 更准确的说法应是：

> 我们不把 feed-to-search transition 先建模成复杂的 end-to-end intent encoder，而是先把它拆成一组廉价、可解释、可实时更新的行为特征；随后构造一个 mode-switch score，将其作为现有排序器的 gating feature，在候选集内动态提升当前 intent 相关内容的权重，而不是直接替换召回层。

### Pinterest user journeys 的可借鉴点与边界

Pinterest 那篇 user journeys 文章，对本项目最有参考价值的地方主要有两点，但都需要分清“原文明确做了什么”和“本项目可以合理借用什么”。

#### 1. 它提供了一个很实用的后排序去重思路

文章在 journey ranking 之后，额外做了一步 diversification。
它不是把相似 topic / journey 直接删除，而是：

- 先得到一批已经排过序的候选
- 再从前往后扫描
- 如果当前候选与前面更高排位的候选过于相似，就对它施加惩罚项
- 最后按更新后的分数重新排序

原文里，这个相似度是用预训练关键词 embedding 来衡量的；当前候选与前面“撞车”的次数越多，分数压得越低；文中提到 penalty 超参数通常取 `0.95`。

对本项目来说，这个思路最值得借用的不是某个具体公式，而是：

> 用一个轻量的“排序后相似项惩罚”机制，避免最后展示出来的前几个 topic / 内容都挤在同一个意思上

翻译到当前项目中，可以直接落成：

- 先得到若干 topic 或 item 候选
- 按基础分排序
- 再从前往后扫描
- 如果后面的候选与前面已保留候选过于相似，就降权，而不是直接删除

这样做的好处是：

- 保留主题信号，不会因为硬删除而损失覆盖
- 让最终结果更有多样性
- 实现非常轻，适合 `V1`

#### 2. 它没有直接给出 passive-to-active 的 mode switch 判别器

文章明确写出来的是另一条路线：

- 先把用户近期行为组织成若干 journey cluster
- 再做 ranking
- 再做 diversification
- 再做 stage prediction

其中，stage prediction 的重点是：

- 区分 `situational` 和 `evergreen`
- 对 `situational` journey 再判断是 `ongoing` 还是 `ended`

原文也明确说，这些 inferred journeys 可以用于 downstream retrieval and ranking。

这对当前项目真正有价值的启发是：

> 可以先构造一个比普通兴趣画像更高一层的中间状态表示，再把这个状态提供给后续的 retrieval / ranking

但同时必须明确：

- Pinterest 公开文章没有直接给出一个“从 passive browsing 切到 active problem solving”的专门判别模块
- 它也没有直接提供一个可拿来即用的 feed-to-search mode switch 模型
- 因此，这部分不能被表述成“直接复现 Pinterest”

更准确的说法应是：

> Pinterest 证明了“先做中间层状态表示，再服务下游召回和排序”这条路是成立的；而本项目的 mode switch 定义，需要结合 ZhihuRec 的 query、click、topic 和 timestamp 自行设计

#### 3. 本项目如何合理借用这个思路

基于这篇文章，本项目更合理的做法不是把 search 当成单个更强 engagement feature，而是把下面这些现象一起看作 mode switch 的候选信号：

- `search` 占比上升
- topic 集中度升高
- 短时间内重复围绕同一 topic 互动
- query 变得更具体、更 task-oriented

一旦检测到这种切换，系统就可以：

- 临时重构推荐时使用的用户状态表示
- 让当前 task / topic 的权重暂时压过长期兴趣
- 在召回和排序阶段提高与当前 task 相关候选的优先级

这样做与 Pinterest 的共同点在于：

- 都不是只盯住单次 engagement
- 都先构造一个高于普通兴趣画像的中间层表示
- 都把这个中间层表示提供给 downstream retrieval / ranking

不同点在于：

- Pinterest 公开讲清楚了 diversification 与 stage prediction
- 但没有公开给出一个专门针对 passive browsing 到 active problem solving 的 mode switch 判别器
- 因此，本项目的这部分设计是借鉴其系统分层思路，而不是照搬其现成模块

### v1 排序边界

`V1` 的排序层应明确采用：

> 规则型轻排序

也就是说：

- 不上单独的学习排序模型
- 不做两阶段复杂排序
- 排序逻辑应保持可解释、可调试、可展示

一个推荐的规则打分结构可以是：

- 长期兴趣匹配作为基础偏好分
- 短期状态 / topic 匹配作为兴趣修正项
- `hot_or_fresh` 候选天然低于更强个性化候选

这里要明确：

- `recent_queries` 对候选的主影响仍发生在召回阶段
- 排序层只允许使用由搜索行为压缩出的少量状态特征或 `query recall boost` 这类派生调试字段
- 但这不等于把 raw `search_query` 直接作为排序项使用

前端调试信息应尽量能展示每个候选的：

- 基础召回分
- topic 匹配分
- query recall boost
- fallback 标记
- 最终排序分

### 为什么

这样可以让第一版：

- 更简单
- 更容易验证
- 更容易调试
- 更容易解释
- 更容易在面试中讲清每个分数项的 trade-off

一个简单的打分直觉可以是：

- 长期兴趣提供基础偏好信号
- topic 匹配提供用户兴趣修正
- recent-query-to-topic 匹配提供轻量级 intent 增强，而不是直接精排打分

这已经足够构成一个有意义的第一版闭环。

## 12. 在线闭环（v1）

### V1 事件边界

`V1` 只持久化三类核心事件：

- `search_query`
- `recommendation_click`
- `search_result_click`

这样做的目的不是否认其他行为的重要性，而是为了在第一版中用最小复杂度支撑完整闭环：

- 有输入意图
- 有推荐消费反馈
- 有搜索后的强确认反馈

`impression`、`skip`、`like`、`favorite` 等行为可以保留为后续扩展，不作为 `V1` 的核心落盘前提。

### V1 画像更新策略

`V1` 的画像更新应明确采用：

> 同步直写画像 + 保留事件表

也就是说：

- 事件一发生，就直接更新数据库中的用户画像
- 同时保留原始事件表，作为可追溯记录和后续扩展基础

这样做的原因是：

- 第一版实现最简单
- 最容易把闭环讲清楚
- 最容易验证“这次行为如何影响下一次推荐”
- 仍然为后续异步化、重放、离线分析保留空间

因此，`V1` **不要**一开始把重点放在：

- 异步事件消费
- 复杂消息队列
- 画像重算流水线
- 近实时流式更新架构

### 推荐闭环

1. 用户打开 feed
2. 系统读取用户画像
3. 系统通过多路轻召回生成答案候选
4. 如果候选不足 / 缺失，则使用 `hot_or_fresh` 补齐
5. 系统合并候选并执行规则型轻排序
6. 系统返回固定 Top `10` 的带调试字段答案卡片
7. 用户进行点击
8. `recommendation_click` 写入事件表，并同步更新 `recent_clicked_answers` 与 `topic_weights`
9. 下一轮推荐体现新的画像变化

### 搜索闭环

1. 用户输入 query
2. query 被写入 `recent_queries`
3. query 被映射到 topic
4. 搜索服务基于 topic 返回最小可用的 answer 结果列表
5. 后续推荐召回获得 topic-aware boost
6. 如果用户点击搜索结果
7. `search_result_click` 写入事件表，并基于 query 命中 topics 与 answer 自带 topics 的组合，以强于推荐点击的权重同步更新 `topic_weights`
8. 被点击的 answer 同时写入 `recent_clicked_answers`，并以更高权重参与用户向量聚合
9. 用户画像因 `recent_queries` 与搜索点击而发生变化
10. 后续推荐体现这一更强信号

这就是简历 / 面试中最核心的闭环故事。

## 13. 权衡原则

所有决策都应该围绕这个首要目标来评估：

> 让这个项目容易写进简历、容易在面试中解释，也容易围绕系统链路和工程权衡展开讨论

### 因此，优先选择：

- 简单但可解释的方案
- 真实但不过度堆砌的架构
- 可见的闭环，而不是复杂且隐藏起来的“高级感”
- 调试可见性，而不是打磨精致的 UI
- 能讲清完整链路与 trade-off，而不是堆积炫技点
- 工程落地优先于训练复杂度
- 同步直写的简单闭环优先于过早异步化
- 单用户可讲清的闭环优先于多用户产品化展示
- 逻辑单体的模块分层优先于过早微服务化
- 先把参数关系定清楚，再通过测试调优具体数值
- 分阶段演进，而不是一口气做成“大而全系统”

### 已经做出的示例权衡

- 以 Answer 为主要推荐实体，而不是更复杂的混合实体 Feed
- 使用轻前端 + 适度模块化的调试前端，而不是构建追求易用性的产品页面
- `V1` 只重建一个小而精的内容世界，而不追求大规模在线数据集
- 先做“多路轻召回 + 状态感知重排 + hot_or_fresh 补齐”，而不是重召回建模
- item 侧优先保证离线 topic / tag / 质量特征产出；embedding 只作为可选增强位
- `V1` 采用“item 离线产物 + user 规则聚合 + 在线轻量重排”，而不是完整重训练双塔
- 用户向量由 `recent_clicked_answers` 加权平均与 `topic_weights` 原型向量构成
- 排序层先采用规则型轻排序，而不是额外引入学习排序模型
- 搜索先做成“极简但真实的召回链路”，而不是完整搜索产品
- 搜索结果以相关性链路为主，热门只做补齐，而不是直接混合加权
- `V1` 只落盘 `search_query / recommendation_click / search_result_click` 三类核心事件
- `V1` 采用“同步更新画像 + 保留事件表”，而不是先做复杂异步事件架构
- `V1` 用户画像先采用单表 + 结构化字段存储，而不是拆很多张画像明细表
- `topic_weights` 采用“加法累积 + 全局衰减”，而不是只增不减或复杂在线学习
- 默认冷启动画像与个性化画像采用线性混合，而不是一刀切切换
- 冷启动混合权重由 `search_query / recommendation_click / search_result_click` 的综合行为分数驱动
- `behavior_score -> alpha` 只定义相对趋势，具体参数通过测试与工程调优确定
- `V1` 的参数调优验证优先采用离线回放测试，而不是只靠人工观察
- `V1` 保留原生 topic 体系，但只做轻清洗并选择一个可控子集
- `V1` 只围绕一个演示用户跑闭环，不做完整多用户系统
- `V1` 后端采用“逻辑单体 + 模块分层”，不引入消息队列和服务注册
- 使用 `topic_weights + recent clicks + recent queries`，而不是庞大的用户画像设计
- 先存原始 query，只有在搜索点击后才更新 topic
- 用共现统计 + 倒排查找做 query-topic 映射，而不是在线 embedding
- `V1` 运行时以 MySQL 作为唯一真源，离线产物只负责导入与重建，Redis 在第一版跑通后再加
- 第一批目录集合固定为 `data/`、`build/demo_world/`、`plan/`、`scripts/`、`sql/`、`backend/`、`frontend/`；其中 `build/demo_world/` 是离线导入包目录，不是运行时目录
- `V1` 只要求本地单机跑通，但必须提供一键初始化脚本
- 先跑通功能闭环，再做延迟优化

### V1 运行与交付边界

`V1` 的运行方式应明确采用：

> 本地单机跑通

也就是说：

- 后端服务、本地数据库、前端页面在本机环境即可完成启动和演示
- 第一版的 MySQL 通过 docker compose 起一个本机容器（见仓库根 `docker-compose.yml`，镜像 `mysql:8.0`），用于避免污染本机环境与跨平台一致性。应用本身（FastAPI 后端、静态前端）不容器化，仍直接用本机 Python 与静态服务器跑。
- 第一版不要求远端部署可访问

第一批真正开始编码时，应明确采用：

> 后端闭环与极简调试前端同阶段推进

也就是说：

- 第一批交付**包含前端**，不把前端整体后置到下一阶段。
- 但前端的职责不是单独追求产品完成度，而是消费后端接口、暴露调试信息、支撑链路验证与演示。
- 如果前后端实现范围发生冲突，应优先保证后端链路、数据边界和调试能力完整，前端只做到“足够服务后端”。

第一批实现的最小目录集合，应明确采用：

> `data/` + `build/demo_world/` + `plan/` + `scripts/` + `sql/` + `backend/` + `frontend/`

也就是说：

- `data/` 继续保存 ZhihuRec 原始数据。
- `build/demo_world/` 继续作为离线派生产物与导入包目录存在，不改造成运行时直读目录。
- `plan/` 继续负责执行拆解，而不是重新定义高层边界。
- `scripts/` 继续承载离线构建、导入、初始化、重置和回放脚本。
- `sql/` 在第一批中立即创建，用来定义 MySQL 作为运行时唯一真源的 schema。
- `backend/` 在第一批中立即创建，用来承载逻辑单体后端及 `recommendation / search / profile / event` 等模块。
- `frontend/` 在第一批中立即创建，但只服务后端调试、验证和演示。
- 第一批**不要求**单独创建 `docs/`；高层边界先统一收敛在本文件，避免 API 契约和边界说明散落到多处。

但同时必须满足：

> 要有一键初始化脚本

这个脚本至少应覆盖：

- 基础环境准备说明或自动化步骤
- 数据库初始化
- 必要的种子数据 / 演示数据准备
- 启动前所需的最小预处理步骤
- 默认演示用户或演示状态重置
- 后端服务启动
- 前端页面启动

也就是说：

- 一键初始化脚本不只负责“把数据准备好”，还要负责把本地演示环境真正拉起。
- 用户执行一次脚本后，应能直接进入可调试、可演示的前后端运行状态。
- 如果某些启动步骤因本地环境差异需要分平台处理，也应由同一入口脚本统一编排，而不是让用户手工拼接多个命令。

此外，`V1` 还应提供：

> 开发脚本形式的一键重置能力

也就是说：

- 支持通过开发脚本重置单用户画像或演示状态
- 这项能力主要服务于调试、离线回放和面试演示
- 不需要在前端提供专门的“重置”入口

在搜索输入这件事上，也应采用同样思路：

- 优先提供开发脚本或调试脚本来模拟 query 输入
- 不要求依赖前端手工操作才能验证搜索链路
- 这类脚本应同时支持单 query 调试和 query 序列回放

这样做的原因是：

- 最符合当前“先跑通、先能演示”的目标
- 最能降低环境搭建摩擦
- 最方便后续自己反复重建环境和快速演示

### V1 API 边界

`V1` 的接口粒度应至少收敛为：

> `/feed` + `/search` + 拆开的事件接口

也就是说：

- `/feed` 默认负责返回最终推荐结果；当 `debug=true` 时，额外返回 recall candidates 与重排轨迹，便于调试召回 / 排序链路
- `/search` 一次请求内同时负责处理 query、记录 `search_query` / `recent_queries`，并返回搜索结果；当 `debug=true` 时，额外返回命中 topics、结果来源、score 组成与 fallback 补位信息
- `search_query` 不再额外走统一事件上报，也不拆单独的前置记录接口，而是在 `/search` 中自然完成写入
- 点击类行为拆成更直白的接口，例如：
  - `/event/recommendation_click`
  - `/event/search_result_click`
- 点击接口默认只返回 `ok`；当 `debug=true` 时，额外返回本次事件导致的画像更新摘要
- 额外提供只读调试接口 `/debug/profile`，用于查看当前 `topic_weights / recent_clicked_answers / recent_queries / behavior_score`，并返回用户向量摘要与最近 `N` 次画像变更摘要

这样做的原因是：

- 第一版调试更直接
- 接口语义更清楚
- 比统一 `event_type` 上报更方便演示和排查

第一版不继续拆更多面向产品化的细接口，先把最小闭环接口链路跑通。

在用户身份处理上，`V1` 应进一步明确为：

> 接口层仍保留 `user_id` 参数，但默认只围绕一个演示用户运行

也就是说：

- API 形态上允许传入 `user_id`
- 但第一版不做真正的多用户产品化使用
- 不做登录系统
- 不做 JWT 或完整鉴权链路

这样做的原因是：

- 保留一定工程合理性
- 但不把项目重心带到认证、会话和安全体系上
- 更符合“这些不是当前项目核心价值”的边界

### 给 Codex 的重要指令

不要过度设计系统。
除非某个额外的大组件能明显增强以下价值，否则不要加：

- 简历价值
- 闭环清晰度
- 面试可解释性

始终优先：

- 最小可运行的端到端流水线
- 明确的权衡
- 易于调试的实现
- 清晰的未来扩展点
- 先把工程链路跑通，再考虑是否补充训练复杂度
- 先用单服务把模块边界定义清楚，再决定未来是否需要拆分
- 先用 MySQL 跑通，再决定是否需要 Redis 优化
- 先提供本地一键初始化与启动能力，再考虑容器化和部署
- 关键参数先做成可调，再通过测试和工程观察收敛

## 17. V1 边界总结

### 一句话版本

`V1` 的本质是：

> 一个面向简历过筛与面试讲述的、强工程闭环但极简前端的内容推荐系统；工程主线是推荐召回链路，故事 hook 是“搜索作为高意图信号反哺推荐”。

### 执行边界

- 项目目标：优先服务面试型工程项目，而不是研究型 Demo 或产品化 UI。
- 工程主线：`A` 是推荐主链路与向量召回工程落地，`B` 是“搜索反哺推荐”的面试故事。
- 推荐架构：`多路轻召回 + 状态感知重排 + hot_or_fresh 补齐 + 规则型轻排序`。
- 候选规模：多路轻召回先产生几十到几百个候选，再排序压到 Top `10`。
- fallback 来源：`hot_or_fresh` 明确采用全局热门 / 新鲜 answer。
- 热门定义：基于 ZhihuRec 的全局交互热度，并截取一个固定时刻的快照来模拟线上状态。
- fallback 触发：只要主召回数量不足 Top `10`，就用 `hot_or_fresh` 补齐。
- 表示策略：item 侧离线生成 topic / tag / 可选 embedding，user 侧采用规则聚合；不做完整重训练双塔。
- Answer 向量：由 `question_title + answer_summary + topics` 生成。
- User 向量：`recent_clicked_answers` 加权平均为主，`topic_weights` 原型向量为辅；最近点击带时间衰减。
- 搜索定位：搜索是极简但真实的召回链路，核心是 `query -> topic -> answer`，不是完整搜索产品。
- 搜索输入：不在输入 UI 上过度投入，优先提供方便后端模拟真实 query 输入的脚本化方式。
- 搜索调试脚本：优先提供命令行脚本，既支持单 query 调试，也支持 query 序列回放。
- 回放输入格式：优先采用可携带时间和动作类型的 JSON 序列。
- 回放动作粒度：至少支持 `search_query / search_result_click / recommendation_click`。
- query-topic 选取：采用配置文件驱动的动态阈值，配置形式优先是“分数阈值 + 最大 topic 数”，而不是写死固定 Top N。
- query-topic score：优先使用离线统计得到的分数聚合，并允许少量规则加成，不走在线 embedding 余弦相似度。
- query-topic 规则：`V1` 只保留“完整短语优先”和“多 token 一致指向加成”两条规则。
- 搜索结果：相关性链路为主，热门只做补齐，不做等价混合加权。
- 搜索结果数：固定返回 Top `10`。
- 搜索调试模式：默认只返回搜索结果；当 `debug=true` 时，额外返回命中的 topics、结果来源、score 组成与 fallback 补位信息。
- 搜索信号用法：raw `search_query` 不直接进入粗排 / 精排；搜索信号优先影响召回层，并可压成少量状态特征进入排序 gating。
- 推荐结果数：固定返回 Top `10`。
- 画像结构：单表 `user_profile`，包含 `topic_weights`、`recent_clicked_answers`、`recent_queries`。
- 画像规模：`topic_weights` 只保留 Top 10；`recent_clicked_answers` 只保留最近 10 条；`recent_queries` 只保留最近 5 条。
- 事件边界：只落盘 `search_query`、`recommendation_click`、`search_result_click`。
- 信号强弱：`search_query < recommendation_click < search_result_click`。
- 画像更新：同步直写画像，同时保留事件表；`search_result_click` 对 `topic_weights` 和用户向量的影响都强于 `recommendation_click`。
- 推荐点击更新：直接依据被点击 answer 自带的 topics 更新 `topic_weights`。
- 搜索点击更新：组合使用 query 命中 topics 与被点击 answer 的 topics 更新 `topic_weights`，其中重合部分视为更强确认。
- 交集处理：只规定“重合部分加得更多”，具体倍数留给配置与调优，不在 brief 中写死。
- `topic_weights` 更新：采用“加法累积 + 全局衰减 + Top 10 截断”。
- 冷启动：默认画像与个性化画像线性混合，混合权重由三类核心事件构成的 `behavior_score` 驱动。
- Topic 空间：保留 ZhihuRec 原生 topic，但只做轻清洗 / 轻合并，并选取一个可控子集。
- 数据规模：只重建一个小而精的内容世界，不追求大规模在线数据集。
- 用户范围：只围绕一个演示用户跑完整闭环，不做登录和多用户系统。
- 第一批交付：包含后端闭环与极简调试前端；前端与后端同阶段推进，但定位明确从属于后端开发、调试、验证与演示。
- 第一批目录：固定为 `data/`、`build/demo_world/`、`plan/`、`scripts/`、`sql/`、`backend/`、`frontend/`；`docs/` 不作为第一批必建目录。
- `build/demo_world/` 定位：保存离线派生产物和导入包，只服务初始化、导入、重建、校验，不参与在线请求直读。
- 前端形态：轻前端 + 适度模块化，单页不是硬约束；职责是服务后端开发、调试、演示，而不是追求易用性。
- Answer 卡片内容：`V1` 只保留标题、摘要、author、topics 四类核心展示信息。
- Feed 调试模式：默认只返回最终 Top `10`；当 `debug=true` 时，额外返回 recall candidates、重排轨迹，以及每条结果的关键打分字段。
- Feed 调试字段：至少显示召回来源、基础召回分、topic 匹配分、query recall boost、最终分、fallback 标记、被选中原因。
- 点击接口调试模式：默认只返回 `ok`；当 `debug=true` 时，返回本次 `topic_weights` / `recent_clicked_answers` / `recent_queries` 的更新摘要，而不是直接返回新 feed 或完整画像快照。
- 画像调试接口：提供只读 `/debug/profile`，明确作为调试能力存在，不伪装成产品接口；返回当前 `topic_weights / recent_clicked_answers / recent_queries / behavior_score`，并可附带用户向量摘要，如维度、范数、top contributing answers/topics，但不直接暴露完整 embedding 数组；同时返回最近 `N` 次画像变更摘要，而不只是一条最近更新。
- 被选中原因：采用规则模板生成，依据最高贡献项或主要贡献组合输出简洁解释。
- 后端形态：逻辑单体 + 模块分层，至少分出 `recommendation / search / profile / event`，不引入消息队列和服务注册。
- API 边界：至少提供 `/feed`、`/search`、`/debug/profile`，以及拆开的点击事件接口；`/feed` 通过 `debug=true` 暴露召回与重排调试信息，`/search` 在一次请求内自然记录 `search_query` / `recent_queries` 并在 `debug=true` 时返回搜索调试信息，点击接口在 `debug=true` 时返回画像更新摘要，`/debug/profile` 负责只读查看当前画像状态、用户向量摘要与最近 `N` 次画像变更摘要。
- 用户身份：接口保留 `user_id` 参数，但 `V1` 不做登录、JWT 和完整鉴权，只围绕默认演示用户运行。
- 运行时真源：`V1` 只以 MySQL 作为在线服务唯一真源；离线产物只负责初始化、导入、重建与校验，不作为运行时直读源。
- 基础设施：`V1` 先只用 MySQL；Redis 等第一版跑通后再加。
- 运行方式：本地单机跑通，必须提供一键初始化脚本，且该脚本覆盖环境检查或说明、MySQL 初始化、demo world 导入、默认演示用户重置、后端启动、前端启动。
- 重置能力：提供开发脚本形式的一键重置能力，不做前端入口。

### 参数与验证

- 参数原则：先定义相对关系，不在 brief 中写死所有公式与倍数；关键参数做成可调。
- 调优方式：优先采用离线回放测试，而不是只靠人工观察。
- 测试场景：至少覆盖推荐点击、搜索不点击、搜索点击三类场景。
- 主优化目标：响应性与稳定性的平衡。
- 关键次指标：`Search Carryover Gain@K`，用于支撑“搜索反哺推荐”的核心故事。
- 面试口径：参数不是拍脑袋定的，而是通过回放测试、代理指标和工程观察收敛得到的。
