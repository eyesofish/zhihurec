# Subproblem 2: 用户决策问答

## 1. Goal
通过多轮、一问一答的方式，把真正会影响 V1 落地方式的关键决策问清楚，并把答案收敛成后续执行输入。

## 2. Why this step exists
当前仓库的方向已经不缺愿景描述，真正缺的是少数几个会改变实现路径的决定。如果这些问题不先问清楚，后面的 `schema`、离线构建脚本和 API 合同都会反复返工。

## 3. Files involved
- `project_brief.md` - 用来约束问题范围，避免问到 brief 已经明确回答过的内容。
- `plan/zhihurec-project-bridge/README.md` - 用来对齐上一轮已经拆出的三大实现步骤。
- `plan/zhihurec-v1-clarification/README.md` - 用来维护当前阶段状态。
- `plan/zhihurec-v1-clarification/02-user-decision-questions.md` - 本文件，记录本轮问答要解决的阻塞项。

## 4. Exact changes
- 只问会改变实现边界的问题，不问已经由 brief 明确给出的高层目标。
- 先问最影响文件结构和技术选型的问题，再问局部实现细节。
- 当前优先要问清的阻塞项包括：
- V1 第一阶段到底要交付“只有后端与离线链路”，还是“后端加极简前端演示台”。
- 运行时存储边界是什么：先以文件产物驱动，还是直接落到数据库。
- 计划借鉴的现有项目骨架是什么，以及这次要复用到什么程度。
- 在线服务优先要实现哪些接口，哪些只需要文档，不需要立即编码。
- 每轮尽量只推进一个核心决策，避免把多个未决问题绑在一起回答。

## 5. Out of scope
- 讨论远期的微服务、消息队列、线上部署和高并发优化。
- 细化模型训练、向量检索或复杂排序算法。
- 直接改旧计划文件里的实现步骤。

## 6. Done condition
已经得到一组足够具体的答案，可以据此决定目录结构、第一批文件、验证命令和演示路径。

## 7. Verification
- 对照阻塞项列表，确认每一项都有明确答案，或者被明确标记为暂缓。
- 确认答案能直接转化为“改哪些文件、先做什么、怎么验收”，而不是停留在偏好表达。

## 8. Expected output
一组冻结过的 V1 决策，用于下一步写成真正可执行的实现计划。

## 9. Notes for the next step
下一步要把这些答案映射到真实文件和真实步骤里，并决定是更新 `zhihurec-project-bridge` 还是新建更细的执行计划。

## 10. Risks or ambiguity
如果问题顺序不当，后面会出现前一个答案推翻后一个答案的情况。所以要优先解决最上游、最影响目录和存储边界的决定。

## Current resolution
本步骤已经完成。V1 的关键决策已经收敛为：

- 第一批包含后端闭环和极简调试前端。
- 运行时只以 MySQL 作为在线真源。
- 原始数据和 `build/demo_world/` 只服务离线构建、导入和校验。
- 后端采用 FastAPI 逻辑单体，不在 V1 引入 Redis、消息队列、登录系统或微服务拆分。
- 后续执行不再继续开放式问答，而是进入 `plan/zhihurec-v1-runtime-closed-loop/`。
